# Compliance Alerts — Notification System, Scheduled Jobs, Email

*Wave 2 Architecture Decision | Depends on: data-model-extract, ui-requirements-extract, cross-cutting-extract, database-schema, api-layer, ui-framework, project-structure*

---

## Decision Summary

Unified alert system with 5 alert categories (lease expiry, compliance deadline, ATP exhaustion, arrears, 2307 missing) fed by 2 data sources: a daily pg_cron PL/pgSQL function (`process_daily_alerts()`) and synchronous tRPC middleware triggers. Alerts stored in the `alert` table (already defined in database-schema). In-app notifications via header bell icon dropdown + dashboard widgets. Email via Supabase Edge Function calling Resend API for URGENT/OVERDUE alerts only. No WebSocket/realtime — alerts loaded on page navigation via React Query. Alert dismissal is the only mutation available to both admin and accountant roles.

---

## 1. Alert Categories

Five distinct alert categories, each with its own generation logic and target audience:

| Category | Source | Generated By | Audience | Levels Used |
|----------|--------|-------------|----------|-------------|
| `LEASE_EXPIRY` | Lease state machine (§5.5 cross-cutting) | Daily pg_cron job | Admin | INFO (90d), WARNING (60d), URGENT (30d), OVERDUE (15d = action required) |
| `COMPLIANCE` | Compliance calendar (~43 deadlines/year) | Daily pg_cron job | Admin, Accountant | INFO (30d), WARNING (15d), URGENT (5d), OVERDUE (past due) |
| `ATP_EXHAUSTION` | ATP utilization check | Synchronous on document issuance | Admin | INFO (80%), WARNING (90%), OVERDUE (100% = blocks issuance) |
| `ARREARS` | Tenant balance check | Synchronous on payment recording + daily job | Admin | WARNING (2 months), URGENT (3 months = ejectment ground for controlled) |
| `FORM_2307_MISSING` | 2307 receipt tracking | Daily pg_cron job (20 days after Q-end) | Admin | WARNING (approaching deadline), URGENT (past deadline) |

### 1.1 Alert Level Semantics

| Level | Color Token | Meaning | Email? | Dashboard? | Bell? | Banner? |
|-------|------------|---------|:------:|:----------:|:-----:|:-------:|
| `INFO` | `--color-alert-info` (blue) | Advance notice, no action needed yet | No | Yes | Yes | No |
| `WARNING` | `--color-alert-warning` (yellow) | Action needed soon | No | Yes | Yes | No |
| `URGENT` | `--color-alert-urgent` (orange) | Immediate action required | **Yes** | Yes | Yes | No |
| `OVERDUE` | `--color-alert-overdue` (red) | Past deadline, may incur penalty | **Yes** | Yes | Yes | **Yes** (persistent banner) |

The `AlertBanner` component (defined in ui-framework) renders at the top of the app shell for:
- OVERDUE compliance deadlines
- ATP at 100% utilization (blocks document issuance)
- Electric VAT treatment = PENDING (from app_settings, not the alert table)

## 2. Alert Generation: Daily Scheduled Job

### 2.1 Single PL/pgSQL Function

One PostgreSQL function handles all daily alert generation. Called by pg_cron at 00:05 Manila time (16:05 UTC previous day).

```sql
SELECT cron.schedule(
  'daily-alert-generation',
  '5 16 * * *',  -- 00:05 PHT (UTC+8) = 16:05 UTC
  $$SELECT process_daily_alerts()$$
);
```

**Why one function instead of separate jobs:** The alert categories share the same `alert` table and deduplication logic. A single function avoids race conditions on deduplication checks and keeps the cron schedule simple.

### 2.2 Function Logic: `process_daily_alerts()`

```
FUNCTION process_daily_alerts() RETURNS void:

  -- 1. LEASE LIFECYCLE (from cross-cutting §5.4)
  -- This also handles state transitions (ACTIVE→EXPIRED→MONTH_TO_MONTH)
  -- Alert generation is a side effect of the lifecycle check

  FOR EACH lease WHERE status = 'ACTIVE':
    days_until_expiry = date_end - CURRENT_DATE

    IF days_until_expiry < 0:
      -- Lease has expired: transition state
      INSERT INTO lease_event (lease_id, event_type, triggered_by)
        VALUES (lease.id, 'EXPIRED', 'SYSTEM')
      UPDATE lease SET status = 'EXPIRED' WHERE id = lease.id

    ELSE IF days_until_expiry IN (90, 60, 30, 15):
      -- Approaching expiry: generate alert (deduplicated)
      level = CASE
        WHEN days_until_expiry = 90 THEN 'INFO'
        WHEN days_until_expiry = 60 THEN 'WARNING'
        WHEN days_until_expiry = 30 THEN 'URGENT'
        WHEN days_until_expiry = 15 THEN 'OVERDUE'
      END
      INSERT INTO alert (level, category, title, message, entity_type, entity_id)
        SELECT level, 'LEASE_EXPIRY',
          'Lease expiring in ' || days_until_expiry || ' days',
          'Lease #' || lease.id || ' (' || tenant.name || ', Unit ' || rentable.unit_label || ') expires on ' || lease.date_end,
          'lease', lease.id
        WHERE NOT EXISTS (
          SELECT 1 FROM alert
          WHERE category = 'LEASE_EXPIRY'
            AND entity_id = lease.id
            AND level = level_val
            AND created_at > CURRENT_DATE - INTERVAL '1 day'
        )

  FOR EACH lease WHERE status = 'EXPIRED':
    days_since_expiry = CURRENT_DATE - date_end
    IF days_since_expiry >= 15
       AND NOT EXISTS (SELECT 1 FROM lease_event WHERE lease_id = lease.id AND event_type = 'RECONDUCTION_STARTED'):
      -- Tacit reconduction
      INSERT INTO lease_event (lease_id, event_type, triggered_by)
        VALUES (lease.id, 'RECONDUCTION_STARTED', 'SYSTEM')
      UPDATE lease SET status = 'MONTH_TO_MONTH' WHERE id = lease.id
      INSERT INTO alert (level, category, title, message, entity_type, entity_id)
        VALUES ('WARNING', 'LEASE_EXPIRY', 'Tacit reconduction',
          'Lease #' || lease.id || ' has entered month-to-month (Art. 1670). Guarantor released (Art. 1672).',
          'lease', lease.id)

  FOR EACH lease WHERE status IN ('ACTIVE', 'MONTH_TO_MONTH'):
    IF CURRENT_DATE = anniversary_date(lease):
      INSERT INTO lease_event (lease_id, event_type, triggered_by)
        VALUES (lease.id, 'ANNIVERSARY', 'SYSTEM')
      INSERT INTO alert (level, category, title, message, entity_type, entity_id)
        VALUES ('INFO', 'LEASE_EXPIRY', 'Lease anniversary',
          'Lease #' || lease.id || ' anniversary — escalation check due.',
          'lease', lease.id)

  -- 2. COMPLIANCE DEADLINES
  FOR EACH obligation IN compliance_obligation:
    next_deadline = compute_next_deadline(obligation)
    days_until = next_deadline - CURRENT_DATE

    -- Check if there's already a compliance_deadline record for this period
    -- If not, create one (UPCOMING status)
    INSERT INTO compliance_deadline (obligation_id, deadline_date, status)
      VALUES (obligation.id, next_deadline, 'UPCOMING')
      ON CONFLICT (obligation_id, deadline_date) DO NOTHING

    IF days_until IN (30, 15, 5) OR days_until < 0:
      level = CASE
        WHEN days_until = 30 THEN 'INFO'
        WHEN days_until = 15 THEN 'WARNING'
        WHEN days_until <= 5 AND days_until >= 0 THEN 'URGENT'
        WHEN days_until < 0 THEN 'OVERDUE'
      END
      -- Deduplicated insert
      INSERT INTO alert (...) WHERE NOT EXISTS (same category + entity + level within 1 day)

    IF days_until < 0 AND deadline.status != 'OVERDUE':
      UPDATE compliance_deadline SET status = 'OVERDUE' WHERE id = deadline.id

  -- 3. ARREARS CHECK
  REFRESH MATERIALIZED VIEW CONCURRENTLY tenant_balance;
  FOR EACH row IN tenant_balance WHERE months_in_arrears >= 2:
    level = CASE
      WHEN months_in_arrears >= 3 THEN 'URGENT'  -- ejectment ground for controlled
      ELSE 'WARNING'
    END
    INSERT INTO alert (...) WHERE NOT EXISTS (same category + entity + level within 7 days)

  -- 4. FORM 2307 TRACKING
  -- Run only in the first 25 days after quarter-end
  IF CURRENT_DATE BETWEEN quarter_end + 1 AND quarter_end + 25:
    FOR EACH corporate_tenant WHERE 2307 not yet received for prior quarter:
      days_past_quarter = CURRENT_DATE - quarter_end
      level = CASE
        WHEN days_past_quarter >= 20 THEN 'URGENT'
        WHEN days_past_quarter >= 10 THEN 'WARNING'
      END
      INSERT INTO alert (...) WHERE NOT EXISTS (deduplicated)
```

### 2.3 Deduplication Strategy

Alerts are deduplicated by `(category, entity_type, entity_id, level)` within a time window:

| Category | Dedup Window | Rationale |
|----------|:------------:|-----------|
| `LEASE_EXPIRY` | 1 day | Each milestone (90/60/30/15) fires once |
| `COMPLIANCE` | 1 day | Deadline alerts fire once per threshold |
| `ATP_EXHAUSTION` | Until dismissed | One alert per threshold (80%, 90%, 100%) |
| `ARREARS` | 7 days | Re-alert weekly until resolved |
| `FORM_2307_MISSING` | 7 days | Re-alert weekly during the 25-day window |

Implemented via `WHERE NOT EXISTS` subquery in the insert, not a unique constraint (because the same entity can have multiple alert levels over time).

## 3. Alert Generation: Synchronous Triggers

Some alerts are generated synchronously during tRPC mutations, not by the daily job:

### 3.1 ATP Exhaustion (on document issuance)

When `document.issue` or `billingRun.finalize` (which issues invoices) increments the document sequence:

```ts
// Inside the atomic document number assignment transaction
const newNumber = await incrementDocumentSequence(db, documentType);
const utilization = computeUtilization(newNumber, atp.seriesStart, atp.seriesEnd);

if (utilization >= 100) {
  // Block issuance — throw PRECONDITION_FAILED
  throw new TRPCError({
    code: "PRECONDITION_FAILED",
    message: "ATP exhausted. Register a new ATP before issuing documents.",
  });
}

if (utilization >= 90 && !existsAlert(db, "ATP_EXHAUSTION", atp.id, "WARNING")) {
  await createAlert(db, {
    level: "WARNING",
    category: "ATP_EXHAUSTION",
    title: "ATP 90% exhausted",
    message: `ATP ${atp.atpNumber} (${documentType}): ${remaining} numbers remaining.`,
    entityType: "authority_to_print",
    entityId: atp.id,
  });
}

if (utilization >= 80 && !existsAlert(db, "ATP_EXHAUSTION", atp.id, "INFO")) {
  await createAlert(db, { level: "INFO", ... });
}
```

**100% blocks issuance** — this is the only alert that prevents a business operation. All other alerts are informational.

### 3.2 Arrears Alert (on payment recording)

After `payment.create` or `payment.allocate`, refresh the tenant's balance and check arrears:

```ts
// After payment allocation
const balance = await getUpdatedTenantBalance(db, tenantId);
if (balance.monthsInArrears >= 3) {
  await createAlertIfNotExists(db, {
    level: "URGENT",
    category: "ARREARS",
    title: "3+ months in arrears — ejectment ground",
    entityType: "tenant",
    entityId: tenantId,
    dedupWindow: "7 days",
  });
}
```

This supplements the daily job — the daily job catches arrears that build up between payments, while the synchronous check catches the case where a payment allocation reveals remaining arrears.

## 4. Compliance Deadline Entity

The `compliance_obligation` table (seed data, ~43 rows) defines recurring deadlines. The `compliance_deadline` table tracks specific instances:

### 4.1 ComplianceObligation (Seed Data)

Already defined in database-schema (seed/compliance.ts). Key fields:

| Field | Type | Example |
|-------|------|---------|
| `name` | text | "VAT Return (2550Q)" |
| `form_number` | text | "2550Q" |
| `frequency` | enum | MONTHLY, QUARTERLY, ANNUAL |
| `regulator` | enum | BIR, SEC, LGU, OTHER |
| `deadline_rule` | jsonb | `{ "day": 25, "relative_to": "QUARTER_END" }` |
| `source_processes` | text[] | ["P11", "P12"] |
| `alert_days_before` | integer[] | [30, 15, 5] |
| `is_active` | boolean | true |

### 4.2 ComplianceDeadline (Generated Per Period)

Tracks the status of each specific deadline instance:

```ts
export const complianceDeadline = pgTable("compliance_deadline", {
  id: serial("id").primaryKey(),
  obligationId: integer("obligation_id").notNull().references(() => complianceObligation.id),
  deadlineDate: date("deadline_date").notNull(),
  status: complianceStatusEnum("status").notNull().default("UPCOMING"),
  filedDate: date("filed_date"),             // When admin marks as filed
  filedBy: text("filed_by"),                 // User ID
  notes: text("notes"),                      // Admin notes
  ...timestamps,
}, (table) => [
  unique("unq_compliance_deadline_obligation_date").on(table.obligationId, table.deadlineDate),
  index("idx_compliance_deadline_status").on(table.status),
  index("idx_compliance_deadline_date").on(table.deadlineDate),
]);
```

**Status flow:** `UPCOMING` → `IN_PROGRESS` (admin starts preparing) → `FILED` (admin marks complete). If deadline passes without filing: daily job sets `OVERDUE`.

### 4.3 Deadline Computation

The `deadline_rule` JSONB field encodes how to compute the next deadline from the current date:

```ts
type DeadlineRule =
  | { day: number; relative_to: "MONTH_END" }           // Monthly: Nth day after month-end
  | { day: number; relative_to: "QUARTER_END" }         // Quarterly: Nth day after quarter-end
  | { day: number; relative_to: "QUARTER_END"; offset_months: number }  // e.g., 60th day
  | { day: number; month: number; relative_to: "ABSOLUTE" }  // Annual: specific date
  | { day: number; months: number[]; relative_to: "ABSOLUTE" }  // Semi-annual: specific dates
  | { relative_to: "EVENT"; event: string }              // Event-driven (manual tracking)

// Examples:
// 0619-E (monthly EWT remittance, 10th/15th): { day: 10, relative_to: "MONTH_END" }
// 2550Q (quarterly VAT, 25th after Q-end):   { day: 25, relative_to: "QUARTER_END" }
// 1702Q (quarterly income, 60th after Q-end): { day: 60, relative_to: "QUARTER_END", offset_months: 0 }
// 1604-E (annual EWT alphalist, Mar 1):       { day: 1, month: 3, relative_to: "ABSOLUTE" }
// LIS (semi-annual, Jan 31/Jul 31):           { day: 31, months: [1, 7], relative_to: "ABSOLUTE" }
```

The `compute_next_deadline()` function in the PL/pgSQL job evaluates this JSONB against the current date to find the next upcoming deadline for each obligation.

## 5. Email Notifications

### 5.1 Decision: Resend via Supabase Edge Function

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| Resend API | Simple REST API, generous free tier (3K/month), good Next.js DX | External dependency | **Selected** |
| Supabase Edge Function + SMTP | No external service | SMTP setup more complex, no templates | Alternative |
| SendGrid / Mailgun | Mature, high deliverability | Overkill for 2-user system with ~5-10 emails/month | Rejected |
| No email | Simplest | Misses deadlines if user doesn't log in daily | Rejected |

### 5.2 Email Trigger Architecture

Emails are NOT sent from the PL/pgSQL function (PostgreSQL has no HTTP client). Instead:

1. Daily job inserts alerts into `alert` table with `email_sent = false`
2. A **second pg_cron job** (runs 1 minute after the alert job) calls a Supabase Edge Function
3. The Edge Function queries for unsent URGENT/OVERDUE alerts, sends emails via Resend, and marks `email_sent = true`

```sql
-- Second cron job: 00:06 PHT daily
SELECT cron.schedule(
  'send-alert-emails',
  '6 16 * * *',  -- 1 minute after alert generation
  $$SELECT net.http_post(
    url := current_setting('app.edge_function_url') || '/send-alert-emails',
    headers := jsonb_build_object('Authorization', 'Bearer ' || current_setting('app.service_role_key')),
    body := '{}'::jsonb
  )$$
);
```

**Why pg_cron → Edge Function instead of direct from Next.js:** The alert generation runs inside PostgreSQL. We need a bridge to HTTP. Supabase's `pg_net` extension provides `net.http_post()` from within PL/pgSQL, which can call a Supabase Edge Function. This keeps the entire pipeline inside Supabase infrastructure — no Vercel cron needed.

Add to the existing `alert` table definition:

### 5.4 Edge Function: `send-alert-emails`

```ts
// supabase/functions/send-alert-emails/index.ts
import { Resend } from "resend";

Deno.serve(async (req) => {
  // Verify service role key
  const supabase = createClient(/* service role */);

  // Fetch unsent URGENT/OVERDUE alerts
  const { data: alerts } = await supabase
    .from("alert")
    .select("*")
    .in("level", ["URGENT", "OVERDUE"])
    .eq("email_sent", false);

  if (!alerts?.length) return new Response("No emails to send");

  // Group by level for a single digest email
  const resend = new Resend(Deno.env.get("RESEND_API_KEY"));

  // Fetch admin email from app_settings or auth.users
  const adminEmail = await getAdminEmail(supabase);

  await resend.emails.send({
    from: "TSVJ Backoffice <alerts@tsvj.app>",
    to: adminEmail,
    subject: `[TSVJ] ${alerts.length} alert(s) require attention`,
    html: renderAlertDigest(alerts),
  });

  // Mark all as sent
  await supabase
    .from("alert")
    .update({ email_sent: true, email_sent_at: new Date().toISOString() })
    .in("id", alerts.map(a => a.id));

  return new Response("OK");
});
```

### 5.5 Email Content

**One daily digest email** (not one email per alert). Contains:

```
Subject: [TSVJ] 3 alert(s) require attention

🔴 OVERDUE
  - RPT payment — Las Piñas (2 days overdue)

🟠 URGENT
  - 0619-E filing deadline Mar 10 (5 days)
  - Lease #42 expiring Mar 15 (15 days)

Login to TSVJ Backoffice to take action: https://tsvj.app
```

**Volume estimate:** ~5-10 emails/month. Most days will have 0 URGENT/OVERDUE alerts. Peak around quarter-end when multiple BIR filings converge.

## 9. Implementation Notes for Feature Specs

### 9.1 What Goes Where

- **F0:** Alert table schema, alert router, alert dropdown, alert banner, alert page, shared `createAlert` helper. The infrastructure must exist before any feature generates alerts.
- **P1:** Lease anniversary alerts (generated by daily job, consumed by escalation feature).
- **P5/P6:** ATP exhaustion alerts (synchronous on document issuance / payment receipt).
- **P6:** Arrears alerts (synchronous on payment recording), Form 2307 missing alerts.
- **P10:** Lease expiry alerts (generated by daily job, consumed by portfolio dashboard).
- **P12:** Compliance deadline alerts, compliance calendar page, compliance seed data.

### 9.2 Forward Loop Order Impact

The compliance system spans multiple features. The forward loop order (F0 → P1 → P5 → P6 → P11 → P2+P3+P4 → P8+P9+P10 → P7+P12+P13+P14) means:

1. **F0** sets up alert infrastructure (table, router, UI components)
2. **P5** adds ATP exhaustion alerts on invoice issuance
3. **P6** adds arrears alerts on payment + Form 2307 tracking alerts
4. **P10** adds lease expiry alerts (daily job section for lease lifecycle)
5. **P12** adds compliance deadline alerts (daily job section for compliance) + compliance calendar page + seed data

The daily pg_cron job is deployed with the database setup (F0), but the lease lifecycle section is only populated when P10 is implemented, and the compliance section when P12 is implemented. The function should be designed with feature flags or conditional sections that gracefully handle missing data.

### 9.3 Cache Invalidation

Alert-related invalidation map (extends api-layer §9):

| Mutation | Invalidates |
|----------|------------|
| `alert.dismiss` | `alert.list`, `alert.count` |
| `alert.dismissAll` | `alert.list`, `alert.count` |
| `compliance.updateStatus` | `compliance.list`, `compliance.getCalendar`, `compliance.getUpcoming` |
| `payment.create` | `alert.list`, `alert.count` (if arrears alert created) |
| `billingRun.finalize` | `alert.list`, `alert.count` (if ATP alert created) |
