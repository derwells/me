# Document Generation — PDF Generation & Spreadsheet Exports

*Wave 2 Architecture Decision | Depends on: data-model-extract, ui-requirements-extract, cross-cutting-extract, project-structure, database-schema, api-layer*

---

## Decision Summary

**PDF generation:** `@react-pdf/renderer` for server-side PDF rendering via dedicated Next.js API routes (`/api/pdf/[type]/[id]`). React component templates for each document type (invoice, receipt, contract, billing statement, Form 2307). PDFs generated on-demand — not stored. Batch download via ZIP streaming for "Download All" scenarios.

**Spreadsheet export:** Client-side SheetJS (`xlsx` 0.20.3 from CDN) via dynamic import, as already decided in the api-layer analysis. tRPC export queries return structured JSON; the client formats and triggers download.

## 1. PDF Library Selection

### 1.1 Options Evaluated

| Library | Approach | Server-Side? | Bundle Impact | Chromium Required? |
|---------|----------|:------------:|:-------------:|:-----------------:|
| **@react-pdf/renderer** | React components → PDF | ✅ | ~0 (server only) | No |
| Puppeteer | HTML → headless Chrome → PDF | ✅ | ~0 (server only) | **Yes** (~400MB) |
| PDFKit | Imperative API | ✅ | ~0 (server only) | No |
| pdfmake | Declarative JSON → PDF | ✅ | ~0 (server only) | No |
| jsPDF | Imperative API + html2canvas | ⚠️ Client mainly | ~300KB | No |

### 1.2 Decision: `@react-pdf/renderer`

**Selected: `@react-pdf/renderer` ^4** (latest stable).

**Rationale:**

1. **React component model** — Fits the Next.js stack. Document templates are JSX components using the same mental model as UI components. Developers (and the forward loop) can reason about PDF layout using familiar patterns.

2. **No Chromium dependency** — Runs in pure Node.js. Critical for deployment simplicity on Vercel (serverless functions have a 50MB uncompressed limit) and Railway (avoids multi-hundred-MB Chromium installs). Puppeteer would require a separate container or Chromium layer.

3. **Built on PDFKit** — Gets PDFKit's battle-tested PDF primitives (fonts, images, vector graphics) with a higher-level component abstraction layer on top.

4. **Server-side rendering** — The `renderToStream()` and `renderToBuffer()` APIs work in Next.js API routes. No browser needed — pure server-side generation.

5. **Structured document focus** — The 5 document types (invoice, receipt, contract, billing statement, Form 2307) are all structured business documents with tables, headers, and text. No complex HTML/CSS rendering needed — this is @react-pdf/renderer's sweet spot.

6. **Custom fonts** — Supports `.ttf`/`.otf` font registration for peso sign (₱) and Filipino character support (ñ in names/addresses).

**Why not Puppeteer:** A 2-user backoffice generating <500 PDFs/month doesn't justify the infrastructure complexity of running headless Chromium in production. Puppeteer adds 400MB+ to the deployment, requires specific Node.js/Chromium version management, and has cold-start penalties in serverless environments.

**Why not PDFKit directly:** @react-pdf/renderer wraps PDFKit with a component model. Using PDFKit directly would mean imperative `doc.text(x, y, ...)` calls — harder to maintain and modify than declarative JSX templates.

**Why not pdfmake:** JSON-based layout definitions work, but lack the composability and type safety of React components. No reusable component patterns, no conditional rendering via JSX.

### 1.3 Pinned Version

```
@react-pdf/renderer: ^4
```

Add to `apps/web/package.json` (not a shared package — PDF rendering is an app-level concern).

### 2.1 Document Data Flow

```
┌──────────────────────────────────────────────────────────────┐
│  tRPC Procedure (data preparation)                           │
│  e.g., billingRun.finalize → creates Charges + IssuedDocument│
│  → assigns sequential document number from DocumentSequence  │
└──────────────────────┬───────────────────────────────────────┘
                       │ (data saved to DB)
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  Next.js API Route: /api/pdf/invoice/[id]                    │
│  1. Read IssuedDocument + IssuedDocumentLine from DB         │
│  2. Read related Tenant, Property, ATP data                  │
│  3. Pass structured data to <InvoicePDF /> component         │
│  4. renderToStream() → pipe to HTTP response                 │
│  5. Content-Type: application/pdf                            │
│  6. Content-Disposition: attachment; filename="INV-2026-341" │
└──────────────────────────────────────────────────────────────┘
```

**Key principle:** tRPC mutations create the data (charges, documents, sequences). PDF rendering is a **read-only** operation served by a dedicated API route. This separation keeps tRPC mutations fast and lets PDF rendering happen lazily on download.

## 3. PDF API Route Architecture

### 3.1 Route Structure

```
apps/web/src/app/api/pdf/
├── invoice/[id]/route.ts      # VAT Sales Invoice PDF
├── receipt/[id]/route.ts      # Official Receipt PDF
├── contract/[id]/route.ts     # Lease Contract PDF
├── statement/[id]/route.ts    # Billing Statement PDF
├── form2307/[id]/route.ts     # Supplier Form 2307 PDF
└── batch/route.ts             # Batch download (ZIP)
```

### 3.2 Route Handler Pattern

Each PDF route follows the same pattern:

```ts
// apps/web/src/app/api/pdf/invoice/[id]/route.ts
import { renderToStream } from "@react-pdf/renderer";
import { createSupabaseServerClient } from "@/lib/supabase/server";
import { createDbClient } from "@tsvj/db";
import { InvoicePDF } from "@/components/pdf/invoice-pdf";

export async function GET(
  request: Request,
  { params }: { params: Promise<{ id: string }> },
) {
  // 1. Auth check (same as tRPC — read session from cookies)
  const supabase = await createSupabaseServerClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return new Response("Unauthorized", { status: 401 });

  // 2. Load document data from DB
  const { id } = await params;
  const db = createDbClient(process.env.DATABASE_URL!);
  const doc = await db.query.issuedDocument.findFirst({
    where: eq(issuedDocument.id, parseInt(id)),
    with: {
      lines: true,
      tenant: true,
      lease: { with: { rentable: true } },
      atp: true,
    },
  });
  if (!doc) return new Response("Not Found", { status: 404 });

  // 3. Load company info (lessor data for invoice header)
  const settings = await db.query.appSettings.findFirst();

  // 4. Render PDF
  const stream = await renderToStream(
    <InvoicePDF document={doc} company={settings} />,
  );

  // 5. Return as streaming response
  return new Response(stream as unknown as ReadableStream, {
    headers: {
      "Content-Type": "application/pdf",
      "Content-Disposition": `inline; filename="${doc.documentNumber}.pdf"`,
    },
  });
}
```

### 3.3 Auth on PDF Routes

PDF routes are NOT served through tRPC (tRPC returns JSON, not binary streams). Auth is handled by reading the Supabase session cookie directly (same `@supabase/ssr` pattern used in tRPC context creation).

Role enforcement per document type:

| Route | Auth Level | Rationale |
|-------|:----------:|-----------|
| `/api/pdf/invoice/[id]` | Any authenticated | Both roles can download invoices |
| `/api/pdf/receipt/[id]` | Any authenticated | Both roles can download receipts |
| `/api/pdf/contract/[id]` | Admin only | Contracts are admin-only (P8) |
| `/api/pdf/statement/[id]` | Any authenticated | Both roles view billing statements |
| `/api/pdf/form2307/[id]` | Admin only | Supplier 2307s are admin-only (P14) |
| `/api/pdf/batch` | Any authenticated | Batch download respects per-type rules |

### 3.4 Batch PDF Download (ZIP)

The "Download All" button on the invoice/receipt register generates a ZIP containing individual PDFs.

```ts
// apps/web/src/app/api/pdf/batch/route.ts
import archiver from "archiver";
import { renderToBuffer } from "@react-pdf/renderer";

export async function POST(request: Request) {
  const { type, ids } = await request.json();
  // type: "invoice" | "receipt"
  // ids: number[] — document IDs to include

  // Auth check...

  // Create ZIP archive
  const archive = archiver("zip", { zlib: { level: 6 } });
  const chunks: Buffer[] = [];

  archive.on("data", (chunk) => chunks.push(chunk));

  for (const id of ids) {
    const doc = await loadDocument(id);
    const buffer = await renderToBuffer(
      <InvoicePDF document={doc} company={settings} />,
    );
    archive.append(buffer, { name: `${doc.documentNumber}.pdf` });
  }

  await archive.finalize();

  return new Response(Buffer.concat(chunks), {
    headers: {
      "Content-Type": "application/zip",
      "Content-Disposition": `attachment; filename="${type}-batch.zip"`,
    },
  });
}
```

**Dependency:** `archiver` ^7 — mature ZIP library for Node.js. Add to `apps/web/package.json`.

**Batch size limit:** Cap at 100 documents per batch to avoid timeout. For ~50 tenants, monthly invoice batch is well within this. The route validates `ids.length <= 100`.

### 4.5 Lease Contract Layout (P8)

Unlike invoices/receipts (which are data-driven with fixed layout), contracts are **template-driven with clause assembly**. The approach:

1. `LeaseTemplate` stores a structured template definition (not raw HTML): ordered list of section IDs
2. `LeaseClause` stores individual clauses: title, body text with `{{variable}}` placeholders, `is_mandatory` per regime, `sort_order`
3. At render time, the `<ContractPDF />` component:
   - Selects the template for the lease's regime
   - Assembles clauses in order (mandatory first, then selected optional)
   - Substitutes `{{variables}}` with lease data
   - Adds DST computation summary
   - Renders as a multi-page PDF

**Variable substitution map:**

| Placeholder | Source |
|-------------|--------|
| `{{lessor_name}}` | AppSettings.company_name |
| `{{lessor_tin}}` | AppSettings.tin |
| `{{lessor_address}}` | AppSettings.address |
| `{{tenant_name}}` | Tenant.name |
| `{{tenant_tin}}` | Tenant.tin |
| `{{tenant_address}}` | Tenant.address |
| `{{unit_description}}` | Rentable.name + Room.name + Property.address |
| `{{monthly_rent}}` | Lease.monthly_rate (formatted as peso) |
| `{{lease_term}}` | Computed from date_start to date_end |
| `{{date_start}}` | Lease.date_start (formatted) |
| `{{date_end}}` | Lease.date_end (formatted) |
| `{{deposit_amount}}` | SecurityDeposit.amount (formatted) |
| `{{advance_amount}}` | Lease.advance_rent (formatted) |

## 5. PDF Storage Strategy

### 5.1 Decision: Generate On-Demand (No Storage)

PDFs are **not stored**. Each download request triggers a fresh render from the current database state.

**Rationale:**
1. **Small scale:** ~50 tenants, <500 PDF requests/month. Render time for @react-pdf/renderer is ~200-500ms per document — imperceptible.
2. **Data consistency:** A stored PDF could become stale if underlying data is corrected (e.g., tenant TIN updated). On-demand generation always reflects current data.
3. **No Supabase Storage needed:** Eliminates the need for file storage configuration, cleanup jobs, and storage costs.
4. **Simplicity:** No file lifecycle management, no orphaned files, no storage quotas.

**Exception — Finalized invoices and receipts:** The `IssuedDocument` entity stores all data needed to reproduce the PDF exactly (all 16 mandatory fields are persisted, not looked up from live tenant/property records). This means even on-demand renders of finalized invoices are deterministic and match the original. The data is frozen at issuance time — the PDF template just lays it out.

### 5.2 Cache Consideration

No PDF caching in MVP. If performance becomes an issue (unlikely at this scale), add a short-lived Redis/memory cache keyed by `document_type + id + updated_at`. But for 2 users, this is premature optimization.

### 6.3 CSV vs XLSX Logic

Both formats use the same tRPC query and data. The format choice is a client-side decision:

```ts
// apps/web/src/components/export-button.tsx
async function handleExport(format: "xlsx" | "csv") {
  const XLSX = await import("xlsx");
  const result = await refetch();
  if (!result.data) return;

  const ws = XLSX.utils.json_to_sheet(result.data.rows);
  const wb = XLSX.utils.book_new();
  XLSX.utils.book_append_sheet(wb, ws, result.data.sheetName);

  if (format === "xlsx") {
    XLSX.writeFile(wb, result.data.filename.replace(/\.csv$/, ".xlsx"));
  } else {
    XLSX.writeFile(wb, result.data.filename, { bookType: "csv" });
  }
}
```

The `ExportButton` composed component (from ui-framework decision) renders as a split button: primary action = XLSX, dropdown = CSV (where applicable).

## 7. PDF Preview in Browser

### 7.1 Inline PDF Viewer

The invoice/receipt detail view shows a PDF preview inline (not just a download link). This uses the browser's native PDF viewer via an `<iframe>`:

```tsx
// Invoice/Receipt detail page — PDF preview tab
<iframe
  src={`/api/pdf/invoice/${documentId}`}
  className="h-[800px] w-full rounded border"
  title="Invoice Preview"
/>
```

**Why iframe over a React PDF viewer library:** The browser's native PDF viewer (Chrome PDF Viewer, Firefox pdf.js) is already available, renders PDFs perfectly, and adds zero bundle size. A dedicated viewer library like `@react-pdf/renderer`'s viewer or `react-pdf` (the display library) would add 200-500KB for no benefit in this use case — the admin just needs to see and download the document.

### 7.2 Contract Preview

The contract generation wizard (Step 4: Preview) shows the contract PDF in the same iframe pattern. The preview uses a draft endpoint that accepts the contract data as query params (or a temporary session key) rather than a persisted document ID, since the contract hasn't been saved yet:

```
/api/pdf/contract/preview?leaseId=48&templateId=2&clauseIds=1,3,5,7
```

This renders the contract PDF on-the-fly from the wizard's selected options. Once the user clicks "Generate," the contract data is persisted and the preview URL switches to the permanent `/api/pdf/contract/[id]` route.

### 9.2 Testing Strategy

PDF component tests use `renderToBuffer()` to generate the PDF and verify:
- Buffer is non-empty (PDF was generated)
- Buffer starts with `%PDF-` magic bytes
- Known text content appears in the PDF (use `pdf-parse` for text extraction in tests)

```ts
// __tests__/pdf/invoice-pdf.test.ts
import { renderToBuffer } from "@react-pdf/renderer";
import { InvoicePDF } from "@/components/pdf/invoice-pdf";
import pdf from "pdf-parse";

test("invoice PDF contains all 16 mandatory fields", async () => {
  const buffer = await renderToBuffer(
    <InvoicePDF document={mockDocument} company={mockCompany} />,
  );
  expect(buffer[0]).toBe(0x25); // %
  expect(buffer[1]).toBe(0x50); // P
  expect(buffer[2]).toBe(0x44); // D
  expect(buffer[3]).toBe(0x46); // F

  const { text } = await pdf(buffer);
  expect(text).toContain("VAT SALES INVOICE");
  expect(text).toContain(mockDocument.documentNumber);
  expect(text).toContain(mockCompany.tin);
  expect(text).toContain(mockDocument.tenant.name);
  expect(text).toContain(mockDocument.tenant.tin);
  expect(text).toContain(mockDocument.atp.atpNumber);
});
```

**Test dependency:** `pdf-parse` ^1 (dev dependency only) — extracts text from PDF buffers for assertion. Alternative: snapshot testing of the PDF buffer (brittle but simple).

### 9.3 Contract Template Seeding

The forward loop seeds initial lease templates and clauses during P8 implementation:
- 1 template for CONTROLLED_RESIDENTIAL
- 1 template for COMMERCIAL
- ~10 mandatory clauses (RA 9653 clauses for controlled, standard Civil Code clauses for commercial)
- ~5 optional clauses (special terms, renovation, subleasing)

Templates and clauses are stored in the database (not files) and managed through the Contract Template Manager UI (admin only).
