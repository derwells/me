# P9 — Lease Renewal & Extension

*Feature Spec | Depends on: F0 (Foundation), P1 (Rent Escalation), P7 (Security Deposit Lifecycle), P8 (Lease Contract Generation) | Implementation order: 11 of 15 (MVP Wave 3: P8+P9+P10)*

---

## Summary

Manage lease continuity through three scenarios: formal renewal (new contract, full DST), extension (addendum, DST on extension period only), and tacit reconduction detection (implied month-to-month after 15 days of acquiescence, no DST). Track non-renewal notices, compute deposit adjustments, validate NHSB caps on controlled residential renewals, link successor leases to predecessors, and reuse the P8 contract generation pipeline for renewal/extension documents.

Both tables are added to the existing `contracts.ts` schema file (same file as `lease_event`, `lease_template`, `lease_clause`, etc.).

|--------|------|-------------|-------|
| id | serial | PK | |
| lease_id | integer | NOT NULL FK→lease(id) RESTRICT | |
| notice_type | notice_type enum | NOT NULL | NON_RENEWAL, VACATE, OWNER_USE_REPOSSESSION |

| ra9653_ground | text | | For controlled units: specific Sec. 9 ejectment ground cited |

| notes | text | | |
| created_by | text | NOT NULL | Supabase auth.users.id |
| created_at | timestamptz | NOT NULL DEFAULT now() | |

*Indexes:*
- `idx_non_renewal_notice_lease_id` on `(lease_id)`
- `idx_non_renewal_notice_sent_date` on `(sent_date)`

*Unique constraint:*
- `unq_non_renewal_notice_lease` UNIQUE on `(lease_id)` — at most one active notice per lease

*Notes:*
- Recording a non-renewal notice prevents tacit reconduction. The daily pg_cron job checks for the existence of a `non_renewal_notice` row before auto-transitioning EXPIRED → MONTH_TO_MONTH.
- For controlled units with `OWNER_USE_REPOSSESSION`, the system computes `re_lease_restriction_until = sent_date + 1 year` and creates a COMPLIANCE alert.

**renewal_record**
| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| id | serial | PK | |
| predecessor_lease_id | integer | NOT NULL FK→lease(id) RESTRICT | Old lease |
| successor_lease_id | integer | NOT NULL FK→lease(id) RESTRICT | New lease |
| renewal_type | renewal_type enum | NOT NULL | RENEWAL, EXTENSION, TACIT_RECONDUCTION |
| effective_date | date | NOT NULL | When the renewed/extended term begins |
| old_monthly_rent | numeric(10,2) | NOT NULL | Rent under predecessor lease |
| new_monthly_rent | numeric(10,2) | NOT NULL | Rent under successor lease (same for reconduction) |
| escalation_applied | numeric(5,4) | | Rate applied (e.g., 0.0230 for 2.3% NHSB) — null for no change |
| deposit_adjustment_amount | numeric(10,2) | | Net amount tenant owes/is owed for deposit — null if no adjustment |
| interest_settled | numeric(10,2) | | Accrued interest returned to tenant at renewal (controlled) |
| board_resolution_id | integer | FK→board_resolution(id) SET NULL | Resolution authorizing renewal (corporate lessee) |
| notes | text | | |
| created_by | text | NOT NULL | Supabase auth.users.id |
| created_at | timestamptz | NOT NULL DEFAULT now() | |

*Indexes:*
- `idx_renewal_record_predecessor` on `(predecessor_lease_id)`
- `idx_renewal_record_successor` UNIQUE on `(successor_lease_id)` — each successor has at most one predecessor
- `idx_renewal_record_type` on `(renewal_type)`

*Check constraints:*
- `chk_renewal_old_rent_positive`: `old_monthly_rent > 0`
- `chk_renewal_new_rent_positive`: `new_monthly_rent > 0`
- `chk_renewal_reconduction_same_rent`: `renewal_type != 'TACIT_RECONDUCTION' OR old_monthly_rent = new_monthly_rent` — tacit reconduction preserves original rent (Art. 1670)

*Notes:*
- The `lease.parent_lease_id` column (defined in F0) provides a quick FK from successor → predecessor. The `renewal_record` table adds audit detail (rent changes, deposit adjustments, type classification).
- For TACIT_RECONDUCTION, the `successor_lease_id` is a synthetic lease record created by the system (status = MONTH_TO_MONTH, date_end = NULL or rolling month-end).
- The DST computation for renewals/extensions is stored on `dst_computation` (P8 table) linked to the successor lease. No DST record for tacit reconduction.

### 2.2 Procedure Details

#### `renewal.getExpiringLeases`

```ts
input: z.object({
  daysAhead: z.number().int().positive().max(365).default(90),
  leaseRegime: leaseRegimeEnum.optional(),
  ...paginationInput,
})

output: z.object({
  items: z.array(z.object({
    leaseId: z.number(),
    tenantName: z.string(),
    tenantId: z.number(),
    rentableLabel: z.string(),
    leaseRegime: leaseRegimeEnum,
    dateEnd: z.string(),
    daysUntilExpiry: z.number(),
    currentMonthlyRent: z.string(),
    hasGuarantor: z.boolean(),          // Art. 1672 warning flag
    hasNonRenewalNotice: z.boolean(),   // Notice already sent
    hasPendingRenewal: z.boolean(),     // Successor lease in DRAFT
    isCorporateTenant: z.boolean(),     // Board resolution needed
  })),
  nextCursor: z.number().nullable(),
})
```

**Logic:**
1. Query leases WHERE `status = 'ACTIVE'` AND `date_end BETWEEN today AND today + daysAhead`
2. Join tenant, lease_rentable→rentable for display data
3. LEFT JOIN non_renewal_notice to check if notice exists
4. LEFT JOIN lease (as successor) via parent_lease_id to check pending renewal
5. Order by `date_end ASC` (most urgent first)

#### `renewal.previewRenewal`

```ts
input: z.object({
  predecessorLeaseId: z.number(),
  renewalType: z.enum(["RENEWAL", "EXTENSION"]),
  newMonthlyRent: z.string(),
  newDateStart: z.string(),
  newDateEnd: z.string(),
  escalationType: escalationTypeEnum.optional(),
  escalationParams: escalationParamsSchema.optional(),
  depositMonths: z.number().int().positive().optional(),
  dstPayer: dstPayerEnum.default("LESSOR"),
  boardResolutionId: z.number().optional(),
})

output: z.object({
  // Validation
  warnings: z.array(z.object({
    code: z.string(),
    message: z.string(),
  })),
  errors: z.array(z.object({
    code: z.string(),
    message: z.string(),
  })),

  // DST preview
  dst: z.object({
    monthlyRent: z.string(),
    annualRent: z.string(),
    leaseTermYears: z.number(),
    perYearBreakdown: dstBreakdownSchema,
    dstAmount: z.string(),
    isExtension: z.boolean(),       // Extension: DST on extension period only
  }).nullable(),                     // null only if tacit reconduction (not applicable here)

  // Deposit adjustment
  depositAdjustment: z.object({
    existingDeposit: z.string(),
    accruedInterest: z.string(),
    newDepositRequired: z.string(),
    netFromTenant: z.string(),       // positive = tenant pays; negative = refund to tenant
    interestSettled: z.string(),
  }).nullable(),

  // Rent comparison
  rentComparison: z.object({
    oldMonthlyRent: z.string(),
    newMonthlyRent: z.string(),
    changeAmount: z.string(),
    changePct: z.string(),
    nhsbCapRate: z.string().nullable(),   // For controlled: the applicable NHSB rate
    nhsbMaxRent: z.string().nullable(),   // For controlled: max allowed rent
    isWithinNhsbCap: z.boolean().nullable(),
    thresholdCrossing: z.boolean(),       // true if new rent > PHP 10,000 (controlled → exits rent control)
  }),

  // Board resolution validation
  boardResolution: z.object({
    required: z.boolean(),             // true if lessee is corporate
    provided: z.boolean(),
    inScope: z.boolean().nullable(),   // null if not provided; true/false if provided
    scopeIssues: z.array(z.string()), // e.g., "Rent exceeds max_monthly_rent"
  }),
})
```

**Logic:**
1. Load predecessor lease with tenant, rentables, security deposit (P7)
2. **Controlled residential validation:**
   - Lookup NHSB cap rate for `newDateStart` year
   - Compute `nhsbMaxRent = oldMonthlyRent × (1 + nhsbRate)` using `@tsvj/computations` escalation module (ROUND_DOWN)
   - If `newMonthlyRent > nhsbMaxRent` → add error: `NHSB_CAP_EXCEEDED`
   - If `newMonthlyRent > 10000` and `leaseRegime === 'CONTROLLED_RESIDENTIAL'` → add warning: `THRESHOLD_CROSSING`
   - Validate deposit: `depositMonths ≤ 2` (RA 9653 Sec. 7) or add error: `DEPOSIT_CAP_EXCEEDED`
3. **DST computation:**
   - For RENEWAL: `computeDST(newMonthlyRent, leaseTermYears)` via `@tsvj/computations`
   - For EXTENSION: `computeDST(newMonthlyRent, extensionYears)` where `extensionYears = CEIL((newDateEnd - predecessorLease.dateEnd) / 365)`
4. **Deposit adjustment:**
   - Call `@tsvj/computations` deposit module `computeDepositAdjustment(oldRent, newRent, existingDeposit, accruedInterest, isControlled)`
5. **Board resolution validation:**
   - If `tenant.isCorporate` → `required = true`
   - If `boardResolutionId` provided: load resolution, check `isInForce`, check `maxMonthlyRent >= newMonthlyRent`, check `maxLeaseYears >= termYears`
6. Return all computed data (no persistence)

#### `renewal.executeRenewal`

```ts
input: z.object({
  predecessorLeaseId: z.number(),
  renewalType: z.enum(["RENEWAL", "EXTENSION"]),
  // New lease terms
  newMonthlyRent: z.string(),
  newDateStart: z.string(),
  newDateEnd: z.string(),
  customDueDay: z.number().int().min(1).max(31).optional(),
  escalationType: escalationTypeEnum,
  escalationParams: escalationParamsSchema.optional(),
  penaltyRateMonthly: z.string().optional(),
  gracePeriodDays: z.number().int().optional(),
  depositMonths: z.number().int().positive(),
  advanceMonths: z.number().int().nonnegative().default(0),
  vatRate: z.string(),
  dstPayer: dstPayerEnum.default("LESSOR"),
  boardResolutionId: z.number().optional(),
  notes: z.string().optional(),
})

output: z.object({
  successorLeaseId: z.number(),
  renewalRecordId: z.number(),
  dstComputationId: z.number().nullable(),
  predecessorStatus: z.string(),          // "RENEWED"
  depositAdjustment: z.object({
    netFromTenant: z.string(),
    interestSettled: z.string(),
  }).nullable(),
  warnings: z.array(z.string()),
})
```

**Logic (transaction-wrapped):**

1. **Validate predecessor lease:**
   - Status must be `ACTIVE`, `EXPIRED`, or `MONTH_TO_MONTH` (cannot renew a DRAFT, TERMINATED, or already RENEWED lease)
   - No existing successor (no lease with `parent_lease_id = predecessorLeaseId` in DRAFT/ACTIVE status)

2. **Validate controlled residential constraints:**
   - Run same validation as `previewRenewal` — if any errors, throw `BAD_REQUEST`
   - NHSB cap check, deposit cap check, advance cap check

3. **Determine new lease regime:**
   - If predecessor was CONTROLLED_RESIDENTIAL and `newMonthlyRent > 10000` → successor regime = `NON_CONTROLLED_RESIDENTIAL` (threshold crossing)
   - Otherwise inherit predecessor's regime

4. **Create successor lease:**
   ```sql
   INSERT INTO lease (
     tenant_id, status, lease_regime, date_start, date_end,
     monthly_rent, custom_due_day, escalation_type, escalation_params,
     penalty_rate_monthly, grace_period_days, deposit_months,
     advance_months, vat_rate, parent_lease_id, notes
   ) VALUES (...)
   ```
   - `parent_lease_id = predecessorLeaseId`
   - `status = 'DRAFT'` (activation follows P8 contract generation + milestone workflow)

5. **Create initial RecurringCharge + RecurringChargePeriod** (same pattern as P1/F0):
   - RecurringCharge for RENT with `base_amount = newMonthlyRent`, `effective_date = newDateStart`
   - RecurringChargePeriod with `period_start = newDateStart`, `amount = newMonthlyRent`

6. **Compute and store DST** (RENEWAL or EXTENSION only):
   - For RENEWAL: `computeDST(newMonthlyRent, leaseTermYears)`
   - For EXTENSION: `computeDST(newMonthlyRent, extensionYears)`
   - INSERT `dst_computation` linked to successor lease
   - INSERT `lease_execution_milestone` (DRAFT) for successor lease

7. **Compute deposit adjustment** (if predecessor has a security_deposit):
   - Call `@tsvj/computations` `computeDepositAdjustment`
   - For controlled RENEWAL: settle old deposit (return + interest), collect new deposit
   - For controlled EXTENSION: top-up to new rent level if escalated
   - For commercial: contractual — top-up to new deposit_months × new_rent
   - Store `deposit_adjustment_amount` and `interest_settled` in renewal_record

8. **Create renewal_record:**
   ```sql
   INSERT INTO renewal_record (
     predecessor_lease_id, successor_lease_id, renewal_type,
     effective_date, old_monthly_rent, new_monthly_rent,
     escalation_applied, deposit_adjustment_amount, interest_settled,
     board_resolution_id, notes, created_by
   ) VALUES (...)
   ```

9. **Update predecessor lease status:**
   - SET `status = 'RENEWED'`

10. **Insert lease events:**
    - Predecessor: `LeaseEvent(RENEWED)` with `details = { renewalType, successorLeaseId, renewalRecordId, ... }`
    - Successor: `LeaseEvent(CREATED)` with `details = { renewalType, predecessorLeaseId, renewalRecordId, ... }`

11. **Handle threshold crossing:**
    - If `newMonthlyRent > 10000` and predecessor was CONTROLLED_RESIDENTIAL:
      - Create `LEASE_EXPIRY` alert: "Lease has exited rent control — regime changed to NON_CONTROLLED_RESIDENTIAL"
      - The successor lease already has `lease_regime = NON_CONTROLLED_RESIDENTIAL` (step 3)

12. Return IDs and warnings

#### `renewal.createNotice`

```ts
input: z.object({
  leaseId: z.number(),
  noticeType: noticeTypeEnum,
  noticeMethod: noticeMethodEnum,
  sentDate: z.string(),
  receivedDate: z.string().optional(),
  ra9653Ground: z.string().optional(),
  notes: z.string().optional(),
})

output: z.object({
  id: z.number(),
  reLeaseRestrictionUntil: z.string().nullable(),
})
```

**Logic (transaction-wrapped):**
1. Load lease — must be `ACTIVE` or `EXPIRED` (within 15-day window)
2. Check no existing notice (unique constraint)
3. If `noticeType = OWNER_USE_REPOSSESSION`:
   - `re_lease_restriction_until = sent_date + 1 year`
   - Create `COMPLIANCE` alert: "Cannot re-lease unit until {date} per RA 9653 Sec. 9(c)"
4. If lease is `EXPIRED` and `daysSinceExpiry >= 15`:
   - Reject: "Too late — tacit reconduction has already occurred. Lease is now MONTH_TO_MONTH."
5. INSERT `non_renewal_notice`
6. INSERT `lease_event(NOTICE_SENT)` with notice details

## 3. Computation Logic — `packages/computations/src/renewal.ts`

### 3.1 `validateRenewalTerms`

```ts
export function validateRenewalTerms(input: {
  leaseRegime: LeaseRegime;
  oldMonthlyRent: string;
  newMonthlyRent: string;
  nhsbCapRate: string | null;       // from nhsb_cap_rate for the start year
  depositMonths: number;
  advanceMonths: number;
}): { errors: ValidationError[]; warnings: ValidationWarning[] }
```

**Rules:**
- If `leaseRegime === 'CONTROLLED_RESIDENTIAL'`:
  - `nhsbMaxRent = roundNHSB(oldMonthlyRent × (1 + nhsbCapRate))` — using ROUND_DOWN from escalation module
  - If `newMonthlyRent > nhsbMaxRent` → error: `NHSB_CAP_EXCEEDED`
  - If `depositMonths > 2` → error: `DEPOSIT_CAP_EXCEEDED` (RA 9653 Sec. 7)
  - If `advanceMonths > 1` → error: `ADVANCE_CAP_EXCEEDED` (RA 9653 Sec. 7)
  - If `newMonthlyRent > 10000` → warning: `THRESHOLD_CROSSING` (exits rent control)
- For all regimes:
  - If `newMonthlyRent <= 0` → error: `INVALID_RENT`

### 3.2 `computeDepositAdjustment`

```ts
export function computeDepositAdjustment(input: {
  oldMonthlyRent: string;
  newMonthlyRent: string;
  existingDeposit: string;
  accruedInterest: string;
  isControlled: boolean;
  depositMonths: number;
}): DepositAdjustmentResult
```

**Controlled residential (RENEWAL):**
```
newMaxDeposit = newMonthlyRent × depositMonths (max 2)
refundOld = existingDeposit + accruedInterest
collectNew = newMaxDeposit
netFromTenant = collectNew - refundOld    // can be negative (refund)
interestSettled = accruedInterest
```

**Controlled residential (EXTENSION — top-up):**
```
newMaxDeposit = newMonthlyRent × depositMonths (max 2)
topUp = max(0, newMaxDeposit - existingDeposit)
netFromTenant = topUp
interestSettled = "0"    // interest not settled until final lease end
```

**Commercial:**
```
newDeposit = newMonthlyRent × depositMonths
topUp = max(0, newDeposit - existingDeposit)
netFromTenant = topUp
interestSettled = "0"    // no statutory interest obligation
```

## 4. Business Logic

### 4.1 Three Renewal Scenarios

| Scenario | Document | DST | Deposit | Predecessor Status | Successor Status |
|----------|----------|-----|---------|-------------------|-----------------|
| **Renewal** | New contract (P8 pipeline) | Full DST on new term | Settle old + collect new (controlled); top-up (commercial) | RENEWED | DRAFT → activate via P8 |
| **Extension** | Addendum (P8 pipeline, addendum template) | DST on extension period only | Top-up if rent increased | RENEWED | DRAFT → activate via P8 |
| **Tacit reconduction** | None (auto-detected by system) | None | Carries forward as-is | EXPIRED → MONTH_TO_MONTH (same lease record, status change) | N/A — same lease continues |

### 4.2 Tacit Reconduction Detection (Daily pg_cron Job)

Handled by the existing `process_daily_alerts()` PL/pgSQL function (defined in compliance-alerts analysis, P10 implements). P9 defines the logic that this function executes for reconduction:

```sql
-- Within process_daily_alerts() function:
-- Phase: Tacit reconduction detection

-- Find EXPIRED leases past 15-day window with no notice
FOR rec IN
  SELECT l.id AS lease_id, l.tenant_id, l.date_end, l.lease_regime,
         EXISTS(SELECT 1 FROM non_renewal_notice n WHERE n.lease_id = l.id) AS has_notice
  FROM lease l
  WHERE l.status = 'EXPIRED'
    AND l.date_end <= CURRENT_DATE - INTERVAL '15 days'
    AND NOT EXISTS(SELECT 1 FROM non_renewal_notice n WHERE n.lease_id = l.id)
LOOP
  -- Transition to MONTH_TO_MONTH
  UPDATE lease SET status = 'MONTH_TO_MONTH', updated_at = NOW()
  WHERE id = rec.lease_id;

  -- Insert reconduction event
  INSERT INTO lease_event (lease_id, event_type, event_date, triggered_by, details)
  VALUES (
    rec.lease_id,
    'RECONDUCTION_STARTED',
    rec.date_end + INTERVAL '15 days',
    'SYSTEM',
    jsonb_build_object(
      'daysSinceExpiry', 15,
      'guarantorReleased', true   -- Art. 1672: always true on reconduction
    )
  );

  -- Create alert for admin
  INSERT INTO alert (category, level, title, message, lease_id, created_at)
  VALUES (
    'LEASE_EXPIRY',
    'WARNING',
    'Tacit reconduction occurred',
    format('Lease for tenant %s has entered month-to-month (tacit reconduction per Art. 1670). Guarantor obligations extinguished per Art. 1672.', rec.tenant_id),
    rec.lease_id,
    NOW()
  )
  WHERE NOT EXISTS (
    SELECT 1 FROM alert
    WHERE category = 'LEASE_EXPIRY' AND lease_id = rec.lease_id
      AND title = 'Tacit reconduction occurred'
      AND created_at > NOW() - INTERVAL '7 days'
  );
END LOOP;
```

**Key behavior:**
- Reconduction = status change on the SAME lease (not a new lease record). The lease transitions EXPIRED → MONTH_TO_MONTH.
- `date_end` is NOT updated. The original end date remains for audit purposes. Billing continues using the last RecurringChargePeriod (same rent amount per Art. 1670).
- Guarantor released (Art. 1672) — logged in event details but no guarantor table in MVP. Alert warns admin.

### 4.3 Reconduction Prevention via Notice

Recording a `non_renewal_notice` for a lease prevents the daily job from transitioning it to MONTH_TO_MONTH. The notice must be created:
- Before expiry (proactive), OR
- Within the 15-day post-expiry window (reactive — still valid per *Buce v. Galeon*)
- NOT after day 15 (reconduction already occurred)

For controlled units with `OWNER_USE_REPOSSESSION`:
- 3-month advance written notice required (RA 9653 Sec. 9(c))
- System validates `sentDate ≤ lease.dateEnd - 90 days` or adds a warning
- `re_lease_restriction_until` computed and stored

### 4.4 Renewal Execution → P8 Contract Pipeline

After `renewal.executeRenewal` creates the successor lease in DRAFT status:
1. The successor lease appears in the P8 contract generation wizard's lease selector
2. Admin generates the contract (same P8 flow: select template → review clauses → preview → generate)
3. P8 records milestones (SIGNED → NOTARIZED → REGISTERED → DST_FILED)
4. On the SIGNED milestone, P8 computes the DST filing deadline

The renewal flow does NOT bypass the P8 contract pipeline — it feeds into it. The only difference from a new lease is that the successor lease has `parent_lease_id` set and a `renewal_record` linking it to the predecessor.

### 4.5 Controlled Residential Constraints on Renewal

| Constraint | Rule | Validation |
|-----------|------|------------|
| Rent cap | `newRent ≤ oldRent × (1 + NHSB_cap)` | Error in preview + hard block in execute |
| Deposit cap | `≤ 2 months of new rent` | RA 9653 Sec. 7 — error if exceeded |
| Advance cap | `≤ 1 month of new rent` | RA 9653 Sec. 7 — error if exceeded |
| Threshold crossing | `newRent > PHP 10,000` → exits rent control | Warning in preview, regime change in execute |
| Ejectment grounds | Only RA 9653 Sec. 9 grounds for non-renewal | Informational in notice UI |

### 4.6 Edge Cases

1. **Tacit reconduction for controlled units:** Once reconduction occurs, lease becomes indefinite-period → Art. 1673(1) suspension kicks in → lessor cannot eject on period expiry alone (RA 9653 Sec. 12). System adds `ejectmentRestricted = true` to month-to-month listings and shows alert. This is a one-way trap: a definite-term controlled lease CAN terminate at expiry, but once reconduction occurs, ejectment becomes drastically harder.

2. **Guarantor release (Art. 1672):** On tacit reconduction, guaranty/surety obligations are extinguished. System logs `guarantorReleased: true` in the reconduction event and shows a warning in the month-to-month listing. No guarantor table in MVP — tracked via boolean flag and lease notes.

3. **Threshold crossing on renewal:** Controlled unit at PHP 9,900/month with 2.3% NHSB → PHP 10,127.70 → successor lease regime = NON_CONTROLLED_RESIDENTIAL. Unlocks: no deposit cap, no escalation cap, commercial penalty rules.

4. **Serial tacit reconduction:** Lease on month-to-month for years. Each month is a new "lease period." To formalize, admin executes a RENEWAL from the MONTH_TO_MONTH lease, creating a new fixed-term successor.

5. **Written holdover agreement:** If lessor and tenant sign a written month-to-month agreement (instead of relying on tacit reconduction), this IS a document → DST applies. Admin should use RENEWAL (not EXTENSION) with a short term and flag DST.

6. **NHSB resolution gap:** If no NHSB rate exists for the renewal start year, the system uses the last available rate and shows a warning: "No NHSB rate for {year} — using {lastYear} rate of {rate}%."

7. **Multi-rentable renewal:** If predecessor lease covers multiple rentables, the successor inherits all lease_rentable associations. DST computed on aggregate monthly rent.

8. **Corporate lessee renewal:** Both lessor's and lessee's board resolutions may be needed. System validates lessor-side resolution via `boardResolutionId`. Lessee-side is tracked via notes/document reference (no separate table).

9. **Retroactive renewal:** If parties negotiate after expiry (but before reconduction day 15), successor lease `date_start` should equal predecessor `date_end + 1 day` to avoid a gap. System allows `date_start ≤ today` with a warning.

10. **Deposit carry-forward on reconduction:** Tacit reconduction preserves original terms — deposit carries forward at original amount. No top-up right without a new written agreement. Interest accrual (P7) continues on the same security_deposit record.

## 6. Validation Rules

### 6.1 Renewal Execution Input

```ts
export const executeRenewalSchema = z.object({
  predecessorLeaseId: z.number().int().positive(),
  renewalType: z.enum(["RENEWAL", "EXTENSION"]),
  newMonthlyRent: z.string().refine(v => {
    const d = new Decimal(v);
    return d.greaterThan(0);
  }, "Monthly rent must be positive"),
  newDateStart: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  newDateEnd: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  customDueDay: z.number().int().min(1).max(31).optional(),
  escalationType: escalationTypeEnum,
  escalationParams: escalationParamsSchema.optional(),
  penaltyRateMonthly: z.string().optional(),
  gracePeriodDays: z.number().int().min(0).max(30).optional(),
  depositMonths: z.number().int().positive().max(12),
  advanceMonths: z.number().int().nonnegative().max(12).default(0),
  vatRate: z.string().refine(v => v === "0.00" || v === "0.12", "VAT must be 0% or 12%"),
  dstPayer: dstPayerEnum.default("LESSOR"),
  boardResolutionId: z.number().int().positive().optional(),
  notes: z.string().max(2000).optional(),
}).superRefine((data, ctx) => {
  if (new Date(data.newDateEnd) <= new Date(data.newDateStart)) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: "End date must be after start date",
      path: ["newDateEnd"],
    });
  }
  // Controlled residential constraints are validated server-side
  // after loading the predecessor lease's regime
});
```

### 6.2 Non-Renewal Notice Input

```ts
export const createNoticeSchema = z.object({

  noticeMethod: z.enum(["PERSONAL", "REGISTERED_MAIL", "EMAIL"]),
  sentDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),

  ra9653Ground: z.string().max(500).optional(),
  notes: z.string().max(2000).optional(),

  if (data.receivedDate && new Date(data.receivedDate) < new Date(data.sentDate)) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: "Received date cannot be before sent date",
      path: ["receivedDate"],
    });
  }
});
```
