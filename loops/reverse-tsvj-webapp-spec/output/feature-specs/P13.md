# P13 — Official Receipt / Invoice Data

*Feature Spec | Depends on: F0 (Foundation), P5 (Monthly Billing Generation), P6 (Tenant Payment Tracking) | Implementation order: 14 of 15 (MVP Phase 4)*

---

## Summary

Unified document management layer for the BIR EOPT dual-document framework (RA 11976 / RR 7-2024). Provides browsable registers for VAT Sales Invoices, Official Receipts, and Credit Memos with search, filtering, and batch PDF download. Implements full ATP lifecycle management with utilization tracking, transition workflow, and exhaustion alerts. Adds receipt cancellation workflow with audit trail and EIS compliance status tracking. Document **creation** happens upstream (invoices in P5, receipts in P6, credit memos in P5) — P13 manages the registry, lifecycle, and administrative views.

---

## 1. Data Model

### 1.1 Schema Changes — `packages/db/src/schema/documents.ts`

P13 adds cancellation tracking columns to the existing `issued_document` table (created by P5). No new tables are introduced — all infrastructure exists from F0 (`document_sequence`, `authority_to_print`) and P5 (`issued_document`, `issued_document_line`).

**issued_document — additional columns**
| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| is_cancelled | boolean | NOT NULL DEFAULT false | RECEIPT type only; invoices use credit memos (P5) |
| cancelled_at | timestamptz | | When cancellation occurred |
| cancelled_by | text | | Supabase auth.users.id |
| cancellation_reason | text | | Required when cancelling |

*New indexes:*
- `idx_issued_document_is_cancelled` PARTIAL on `(document_type, issue_date)` WHERE `is_cancelled = true` — cancelled document queries
- `idx_issued_document_eis_status` PARTIAL on `(eis_submission_status)` WHERE `eis_submission_status != 'PENDING'` — EIS tracking queries

*Check constraint:*
- `chk_issued_document_cancellation` — `(is_cancelled = false) OR (cancellation_reason IS NOT NULL)` — reason required for cancellation

### 2.4 Procedure Details

#### `document.listInvoices` / `document.listReceipts` / `document.listCreditMemos`

Paginated list of documents filtered by type. All three share the same query logic with a fixed `document_type` filter.

```
Query: SELECT from issued_document
  JOIN tenant (for display name fallback)
  JOIN authority_to_print (for ATP number)
  WHERE document_type = $type
  + optional filters (tenantId, billingPeriod, dateRange, isCancelled)
  ORDER BY $sortBy $sortOrder
  LIMIT $limit
```

For receipts: join `payment` to include `payment_method`.
For credit memos: join `credit_memo` to include reason and original charge reference.

#### `document.getById`

Full document detail with:
1. All `issued_document` columns
2. All `issued_document_line` rows (ordered by `line_number`)
3. Joined `authority_to_print` for ATP info
4. **Linked documents** — resolved via:
   - RECEIPT: query `issued_document_line.charge_id` → find INVOICE documents containing those charges
   - INVOICE: query `issued_document` WHERE `payment_id IN (SELECT id FROM payment WHERE ...)` to find receipts
   - CREDIT_MEMO: link via `credit_memo.issued_document_id`

#### `document.search`

Cross-type text search across all document types:

```
WHERE (
  document_number ILIKE '%' || $query || '%'
  OR tenant_name_snapshot ILIKE '%' || $query || '%'
  OR tenant_tin_snapshot ILIKE '%' || $query || '%'
)
AND ($documentType IS NULL OR document_type = $documentType)
AND ($dateFrom IS NULL OR issue_date >= $dateFrom)
AND ($dateTo IS NULL OR issue_date <= $dateTo)
ORDER BY issue_date DESC
```

#### `document.cancelReceipt`

Transaction-wrapped receipt cancellation:

```
1. Validate document exists and is RECEIPT type
2. Validate document is not already cancelled
3. Validate the associated payment (if any) is reversed
   — Cannot cancel a receipt for an active payment
   — Payment must be reversed (P6 payment.reverse) BEFORE receipt cancellation
4. UPDATE issued_document SET
     is_cancelled = true,
     cancelled_at = NOW(),
     cancelled_by = ctx.user.id,
     cancellation_reason = input.reason
5. INSERT audit_log (action: "document.cancelReceipt")
6. Return updated document
```

**Rule:** Invoices are NEVER cancelled — use credit memos (P5 `billingRun.createCreditMemo`). Only receipts can be cancelled.

**Rule:** The cancelled receipt retains its document number in the sequence. The number is consumed — gapless numbering means no reuse. The register displays it with a CANCELLED badge.

#### `document.getIssuanceSummary`

Aggregation query for document issuance KPIs:

```
SELECT
  document_type,
  COUNT(*) as count,
  COUNT(*) FILTER (WHERE is_cancelled = true) as cancelled_count,
  SUM(total_base_amount) as total_base,
  SUM(total_vat_amount) as total_vat,
  SUM(total_amount) as total_amount,
  SUM(vatable_sales) as vatable_sales,
  SUM(vat_exempt_sales) as vat_exempt_sales
FROM issued_document
WHERE issue_date BETWEEN $periodStart AND $periodEnd
GROUP BY document_type
```

Plus receipt breakdown by payment method (join `payment`).
Plus ATP utilization snapshot per active ATP.

#### `document.updateEISStatus`

Update EIS submission tracking:

```
1. Validate document exists
2. Validate status transition is valid:
   - PENDING → SUBMITTED (submission_reference required)
   - SUBMITTED → ACCEPTED
   - SUBMITTED → REJECTED (rejection_reason required)
   - REJECTED → SUBMITTED (re-submission)
3. UPDATE issued_document SET eis_submission_status = $status
4. INSERT audit_log
```

**Note:** EIS is future-facing (deadline December 31, 2026 per RR 26-2025). The status field and this procedure prepare the data model for eventual BIR EIS API integration. No actual BIR API calls in MVP.

#### `atp.list`

Paginated ATP list with computed utilization:

```
SELECT atp.*,
  (SELECT COUNT(*) FROM issued_document WHERE atp_id = atp.id) as used,
  atp.series_end - atp.series_start + 1 as total_in_series
FROM authority_to_print atp
WHERE ($documentType IS NULL OR atp.document_type = $documentType)
  AND ($isActive IS NULL OR atp.is_active = $isActive)
  AND ($propertyId IS NULL OR atp.property_id = $propertyId)
ORDER BY atp.is_active DESC, atp.created_at DESC
```

Utilization percentage computed in the router: `(used / totalInSeries) * 100`.

#### `atp.create`

Create a new Authority to Print record:

```
1. Validate no overlap with existing ATP ranges for same document_type + property_id:
   — new.seriesStart must not fall within [existing.seriesStart, existing.seriesEnd]
   — new.seriesEnd must not fall within [existing.seriesStart, existing.seriesEnd]
2. Validate atp_number is unique (UNIQUE constraint on table)
3. INSERT authority_to_print with is_active = false
   — New ATPs start inactive; use atp.transition to activate
4. INSERT audit_log
```

#### `atp.transition`

Transaction-wrapped ATP transition — switches active ATP from exhausted/expiring to new:

```
1. Validate currentAtpId exists and is_active = true
2. Validate newAtpId exists and is_active = false
3. Validate both ATPs have same document_type and property_id
4. Validate newAtpId series range is valid (seriesEnd > seriesStart)
5. In transaction:
   a. UPDATE authority_to_print SET is_active = false WHERE id = $currentAtpId
   b. UPDATE authority_to_print SET is_active = true WHERE id = $newAtpId
   c. UPDATE document_sequence SET
        atp_id = $newAtpId,
        current_number = newAtp.series_start - 1,
        series_start = newAtp.series_start,
        series_end = newAtp.series_end,
        prefix = COALESCE($newPrefix, existing.prefix)
      WHERE document_type = $documentType AND property_id = $propertyId
   d. Dismiss any existing ATP_EXHAUSTION alerts for currentAtpId
   e. INSERT audit_log
6. Return { deactivated, activated, sequence }
```

#### `atp.getUtilization`

Utilization report per document type:

```
For each active ATP (optionally filtered by documentType):
1. Count issued documents: SELECT COUNT(*) FROM issued_document WHERE atp_id = $atpId
2. Compute: used, remaining, utilizationPct
3. Estimate exhaustion date:
   — Average daily issuance rate over last 90 days
   — estimatedExhaustionDate = today + (remaining / dailyRate) days
   — NULL if dailyRate = 0
4. Determine alertLevel:
   — OK: utilizationPct < 80
   — WARNING: 80 ≤ utilizationPct < 90
   — CRITICAL: 90 ≤ utilizationPct < 100
   — EXHAUSTED: remaining = 0
```

## 4. Business Logic

### 4.1 Receipt Cancellation Rules

1. **Only RECEIPT type can be cancelled.** Invoices use credit memos (P5 `billingRun.createCreditMemo`). Credit memos cannot be cancelled.
2. **Payment must be reversed first.** If the receipt is linked to a payment (`payment_id IS NOT NULL`), the payment must have `is_reversed = true` before the receipt can be cancelled. This prevents orphaned receipts — the financial reversal (P6) happens before the document cancellation.
3. **Cancelled receipts retain their document number.** The gapless sequence is preserved — no number reuse. The register shows cancelled receipts with a visual CANCELLED badge.
4. **Cancellation is permanent.** A cancelled receipt cannot be un-cancelled.
5. **Audit trail.** Cancellation records `cancelled_by`, `cancelled_at`, `cancellation_reason` on the document. Also logged in `audit_log`.

### 4.2 ATP Lifecycle

```
┌──────────┐     ┌──────────┐     ┌───────────┐
│ INACTIVE │────►│  ACTIVE  │────►│ EXHAUSTED │
│ (created)│     │ (in use) │     │ (full)    │
└──────────┘     └──────────┘     └───────────┘
                      │
                      │ (if valid_to < today)
                      ▼
                 ┌───────────┐
                 │  EXPIRED  │
                 └───────────┘
```

**States:**
- **INACTIVE:** Created but not yet linked to a document sequence. Awaiting transition.
- **ACTIVE:** Currently linked to the document sequence. Documents being issued from this ATP's series range.
- **EXHAUSTED:** All numbers in the series consumed (`current_number = series_end`). Triggered automatically by `getNextDocumentNumber` when it fails to increment.
- **EXPIRED:** `valid_to` date has passed. Detected by daily pg_cron job. Creates ATP_EXHAUSTION alert.

**Transition rules:**
- Only one ATP can be ACTIVE per `(document_type, property_id)` combination
- Transition atomically swaps: deactivate old → activate new → update sequence
- Old ATP transitions to EXHAUSTED (or remains EXPIRED if that was the trigger)
- New ATP must be INACTIVE with `is_active = false`

### 4.3 ATP Utilization Computation

```ts
// In document router or @tsvj/computations
function computeAtpUtilization(atp: AuthorityToPrint, issuedCount: number): ATPUtilization {
  const totalInSeries = atp.seriesEnd - atp.seriesStart + 1;
  const used = issuedCount;
  const remaining = totalInSeries - used;
  const utilizationPct = (used / totalInSeries) * 100;

  return {
    totalInSeries,
    used,
    remaining,
    utilizationPct: Math.round(utilizationPct * 100) / 100, // 2 decimal places
    isNearExhaustion: utilizationPct >= 80,
    isExhausted: remaining === 0,
  };
}
```

**Estimated exhaustion date:**
```ts
function estimateExhaustionDate(remaining: number, last90DaysCount: number): Date | null {
  if (last90DaysCount === 0) return null;
  const dailyRate = last90DaysCount / 90;
  const daysRemaining = remaining / dailyRate;
  return addDays(new Date(), Math.ceil(daysRemaining));
}
```

### 4.4 Linked Document Resolution

For the "Linked Documents" tab on the document detail page:

**INVOICE → Receipts:**
```sql
-- Find receipts that paid charges from this invoice
SELECT DISTINCT id.*
FROM issued_document id
WHERE id.document_type = 'RECEIPT'
  AND id.payment_id IN (
    SELECT pa.payment_id
    FROM payment_allocation pa
    JOIN issued_document_line idl ON idl.charge_id = pa.charge_id
    WHERE idl.issued_document_id = $invoiceId
```

**RECEIPT → Invoices:**
```sql
-- Find invoices whose charges were paid by this receipt's payment
SELECT DISTINCT id.*

JOIN payment_allocation pa ON pa.charge_id = idl.charge_id
WHERE id.document_type = 'INVOICE'
```

### 4.5 Batch PDF Download

The batch download uses the existing `/api/pdf/batch` route (from document-generation architecture decision):

```ts
// Client-side: collect selected document IDs, POST to batch endpoint
async function handleBatchDownload(selectedIds: number[], documentType: "invoice" | "receipt") {
  const response = await fetch("/api/pdf/batch", {
    method: "POST",
    body: JSON.stringify({ type: documentType, ids: selectedIds }),
  });
  const blob = await response.blob();
  const url = URL.createObjectURL(blob);
  // Trigger download
  const a = document.createElement("a");
  a.href = url;
  a.download = `${documentType}-batch-${format(new Date(), "yyyy-MM-dd")}.zip`;
  a.click();
  URL.revokeObjectURL(url);
}
```

**Batch size limit:** 100 documents max (validated server-side). For monthly billing of ~50 tenants, this is sufficient.

## 5. Validation Rules

### 5.1 Receipt Cancellation

```ts
const cancelReceiptInput = z.object({
  id: z.number().int().positive(),
  reason: z.string()
    .min(10, "Cancellation reason must be at least 10 characters")
    .max(500, "Cancellation reason must be at most 500 characters"),
});
```

Server-side validations (in tRPC procedure):
1. Document must exist: `NOT_FOUND` if missing
2. Document must be RECEIPT type: `BAD_REQUEST` — "Only receipts can be cancelled. Use credit memos for invoices."
3. Document must not be already cancelled: `CONFLICT` — "Receipt is already cancelled"
4. Linked payment must be reversed: `PRECONDITION_FAILED` — "Payment must be reversed before cancelling the receipt. Reverse payment first via Payments."

### 5.2 ATP Creation

```ts
const atpCreateInput = z.object({
  documentType: z.enum(["INVOICE", "RECEIPT", "CREDIT_MEMO"]),
  atpNumber: z.string().min(1).max(30),
  seriesStart: z.number().int().positive(),
  seriesEnd: z.number().int().positive(),
  validFrom: z.string().date(),
  validTo: z.string().date().optional(),
  printerName: z.string().min(1).max(255),
  printerTin: z.string().regex(/^\d{3}-\d{3}-\d{3}-\d{3}$/),
  printerAccreditation: z.string().min(1).max(50),
  propertyId: z.number().int().positive(),
}).refine(
  (d) => d.seriesEnd > d.seriesStart,
  { message: "Series end must be greater than series start", path: ["seriesEnd"] },
).refine(
  (d) => d.seriesEnd - d.seriesStart + 1 >= 100,
  { message: "ATP series must have at least 100 numbers", path: ["seriesEnd"] },
);
```

Server-side validations:
1. ATP number must be unique: `CONFLICT`
2. Series range must not overlap with any existing ATP for the same `(document_type, property_id)`: `CONFLICT` — "Series range overlaps with ATP {existing.atpNumber}"
3. Property must exist: `NOT_FOUND`
4. Printer TIN format validated by Zod regex

### 5.3 ATP Transition

Server-side validations:
1. Current ATP must be active: `PRECONDITION_FAILED`
2. New ATP must be inactive: `PRECONDITION_FAILED` — "New ATP is already active"
3. Both ATPs must have same `document_type`: `BAD_REQUEST` — "Cannot transition between different document types"
4. Both ATPs must have same `property_id`: `BAD_REQUEST` — "Cannot transition between different properties"
5. Document sequence must exist for this `(document_type, property_id)`: `NOT_FOUND`

### 5.4 ATP Update

Server-side validations:
1. ATP must exist: `NOT_FOUND`
2. Cannot change `document_type`, `atp_number`, `series_start`, `series_end` after creation: `BAD_REQUEST` — "Series range cannot be modified after ATP creation"
3. If `validTo` is set, it must be after `validFrom`: `BAD_REQUEST`

### 5.5 EIS Status Update

Server-side validations:
1. Document must exist: `NOT_FOUND`
2. Valid transitions only: `BAD_REQUEST`
   - PENDING → SUBMITTED
   - SUBMITTED → ACCEPTED or REJECTED
   - REJECTED → SUBMITTED
   - ACCEPTED is terminal (no further changes)
3. If REJECTED, `rejectionReason` required: validated by Zod refine

## 8. Acceptance Criteria

### 8.1 Schema (6 criteria)

- [ ] Cancellation columns added to issued_document — `ALTER TABLE issued_document ADD COLUMN is_cancelled boolean NOT NULL DEFAULT false` exists and migration applies cleanly
- [ ] Check constraint enforced — `INSERT INTO issued_document (..., is_cancelled, cancellation_reason) VALUES (..., true, NULL)` returns CHECK violation error
- [ ] Partial index on cancelled documents — `SELECT indexname FROM pg_indexes WHERE tablename = 'issued_document' AND indexname = 'idx_issued_document_is_cancelled'` returns 1 row
- [ ] Partial index on EIS status — `SELECT indexname FROM pg_indexes WHERE tablename = 'issued_document' AND indexname = 'idx_issued_document_eis_status'` returns 1 row
- [ ] `pnpm --filter @tsvj/db drizzle-kit check` exits 0 with no pending changes
- [ ] `pnpm --filter @tsvj/db tsc --noEmit` exits 0

### 8.2 Document Register (10 criteria)

- [ ] Invoice list — `document.listInvoices` with no filters returns paginated invoices sorted by `issue_date DESC` with `nextCursor`
- [ ] Receipt list — `document.listReceipts` with no filters returns paginated receipts with payment method and linked invoice numbers
- [ ] Credit memo list — `document.listCreditMemos` returns credit memos with original invoice reference and reason
- [ ] Tenant filter — `document.listInvoices` with `tenantId` returns only that tenant's invoices
- [ ] Period filter — `document.listInvoices` with `billingPeriod: "2026-03"` returns only March 2026 invoices
- [ ] Date range filter — `document.listReceipts` with `dateFrom` and `dateTo` returns receipts within range
- [ ] Cancelled filter — `document.listReceipts` with `isCancelled: true` returns only cancelled receipts
- [ ] Cross-type search — `document.search` with `query: "INV-2026-034"` returns matching invoices; `query: "Santos"` returns documents for tenant Maria Santos
- [ ] Document detail — `document.getById` returns full document with lines, linked documents, ATP info, and cancellation status
- [ ] Linked documents — For an invoice with a paid receipt, `document.getById` returns the receipt in `linkedReceipts[]`; for the receipt, returns the invoice in `linkedInvoices[]`

### 8.3 Receipt Cancellation (8 criteria)

- [ ] Happy path — `document.cancelReceipt` on a receipt with reversed payment sets `is_cancelled=true`, `cancelled_at` to current time, `cancelled_by` to admin user ID, `cancellation_reason` to provided reason
- [ ] Invoice rejection — `document.cancelReceipt` on an INVOICE returns `BAD_REQUEST` with message "Only receipts can be cancelled"
- [ ] Already cancelled — `document.cancelReceipt` on an already-cancelled receipt returns `CONFLICT`
- [ ] Payment not reversed — `document.cancelReceipt` on a receipt whose payment has `is_reversed=false` returns `PRECONDITION_FAILED` with message mentioning payment reversal
- [ ] Reason too short — `document.cancelReceipt` with reason < 10 characters returns `ZodError` with path `["reason"]`
- [ ] Audit log — After cancellation, `audit_log` contains entry with action `"document.cancelReceipt"` and entity_id matching the receipt
- [ ] Gapless preservation — After cancelling receipt OR-2026-0235, the next receipt issued gets OR-2026-0236 (not 0235)
- [ ] Accountant blocked — `document.cancelReceipt` with accountant session returns `FORBIDDEN`

### 8.4 ATP Management (10 criteria)

- [ ] ATP list — `atp.list` returns ATPs sorted by `is_active DESC, created_at DESC` with computed utilization fields
- [ ] ATP detail — `atp.getById` returns full ATP with monthly usage breakdown and active alerts
- [ ] ATP create — `atp.create` with valid input creates inactive ATP record; `atpNumber` uniqueness enforced (duplicate returns `CONFLICT`)
- [ ] Series overlap prevention — `atp.create` with series range overlapping an existing ATP for same `(document_type, property_id)` returns `CONFLICT`
- [ ] ATP update — `atp.update` changes `validTo`, `printerName`, `printerTin`, `printerAccreditation` but rejects changes to `seriesStart` or `seriesEnd`
- [ ] ATP transition — `atp.transition` atomically deactivates old ATP, activates new ATP, updates document_sequence in a single transaction; verifies both ATPs have same `document_type` and `property_id`
- [ ] Post-transition numbering — After transitioning invoice ATP from (old: series 0001-5000) to (new: series 5001-10000), next invoice gets number 5001
- [ ] Utilization report — `atp.getUtilization` returns correct used/remaining/utilizationPct computed from `COUNT(issued_document WHERE atp_id = $id)`
- [ ] Estimated exhaustion — `atp.getUtilization` with non-zero issuance rate returns `estimatedExhaustionDate`; with zero rate returns null
- [ ] Accountant read-only — Accountant can call `atp.list` and `atp.getById` but `atp.create`, `atp.update`, `atp.transition` return `FORBIDDEN`

### 8.5 Issuance Summary (4 criteria)

- [ ] Monthly summary — `document.getIssuanceSummary` with `period: "monthly", year: 2026, month: 3` returns invoice count, receipt count, credit memo count, and VAT totals for March 2026
- [ ] Quarterly summary — `document.getIssuanceSummary` with `period: "quarterly", year: 2026, quarter: 1` returns aggregated data for Q1 2026
