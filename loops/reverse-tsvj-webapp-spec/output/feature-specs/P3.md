# P3 — Electric Billing

*Feature Spec | Depends on: F0 (Foundation), P2 (shared enums) | Implementation order: 6 of 15 (MVP Wave 2: P2+P3+P4)*

---

## Summary

Apportion the monthly Meralco master meter bill across tenants using sub-meter readings and a blended rate (total bill ÷ total kWh). Allocate common area electricity as a separate line item. Enforce no-markup pass-through (EPIRA RA 9136). Support an optional per-kWh admin fee as a separate line item. Handle the CONFLICTING VAT treatment (configurable per `AppSettings.electric_vat_treatment`). Support shared meters (co-tenancy) and defective meter estimated billing.

#### `electric.saveReadings`

Batch save — upserts readings for all meters in one call. Used by the meter reading entry form.

```ts
input: z.object({
  billingPeriod: z.string().regex(/^\d{4}-\d{2}$/),
  readings: z.array(z.object({
    electricMeterId: z.number().int().positive(),
    readingDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
    readingValue: z.string().regex(/^\d+(\.\d{1,2})?$/),
    isEstimated: z.boolean().default(false),
  })).min(1),
})
```

Validation:
- Each meter must exist and be ACTIVE or DEFECTIVE
- `readingValue` must be non-negative
- If a reading already exists for this meter+period, overwrite it (upsert)
- Does NOT enforce reading ≥ previous (handled as warning in UI, confirmed by user)

#### `electric.createMeralcoBill`

```ts
input: z.object({
  propertyId: z.number().int().positive(),
  billingPeriod: z.string().regex(/^\d{4}-\d{2}$/),
  billingPeriodStart: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  billingPeriodEnd: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  totalKwh: z.string()
    .regex(/^\d+(\.\d{1,2})?$/)
    .refine((v) => parseFloat(v) > 0, "Total kWh must be positive"),
  totalAmount: z.string()
    .regex(/^\d+\.\d{2}$/, "Must be amount with 2 decimals")
    .refine((v) => parseFloat(v) > 0, "Total must be positive"),
  billComponents: meralcoBillComponentsSchema,
  billNumber: z.string().max(50).optional(),
  dueDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional(),
}).refine(
  (data) => data.billingPeriodStart < data.billingPeriodEnd,
  "Start date must be before end date"
)
```

**Computed on save:** `blended_rate_per_kwh = roundElectricRate(total_amount / total_kwh)` — stored for reference.

Returns `CONFLICT` if a bill already exists for this property+period.

#### `electric.run`

Computes all electric charges for a billing period. Transaction-wrapped.

```ts
input: z.object({
  propertyId: z.number().int().positive(),
  billingPeriod: z.string().regex(/^\d{4}-\d{2}$/),
  allocationMethod: z.enum(["FLOOR_AREA", "EQUAL_SPLIT"]),
  adminFeePerKwh: z.string()
    .regex(/^\d+\.\d{4}$/)
    .refine((v) => parseFloat(v) >= 0, "Admin fee must be non-negative")
    .default("0.0000"),
})
```

**Preconditions checked (returns PRECONDITION_FAILED if not met):**
1. Meralco bill exists for this property+period
2. All ACTIVE sub-meters have readings for this period
3. No FINALIZED run exists for this property+period (a DRAFT run is deleted and recreated)
4. `electric_vat_treatment` setting is NOT `"PENDING"` — must be explicitly configured before running (returns PRECONDITION_FAILED with message: "Electric VAT treatment must be configured in Settings before running billing")

**Process:**
1. Delete existing DRAFT run for this period (if any)
2. Fetch Meralco bill, all readings, `electric_vat_treatment` setting
3. Compute `blendedRate = roundElectricRate(meralcoBill.totalAmount / meralcoBill.totalKwh)`
4. Determine `vatRate`: if setting = `"VATABLE"` → `"0.12"`, if `"EXEMPT"` → `"0.00"`
5. For each SUB_METER with ACTIVE status and tenant assignment:
   a. Get current + previous readings; compute consumption delta (kWh)
   b. Call `@tsvj/computations` `computeElectricBilling()` with consumption, blendedRate, adminFeePerKwh, vatRate
   c. Handle shared meters: compute full charge, then apply `share_pct / 100`
   d. Create `electric_charge` row
6. For COMMON_AREA meters:
   a. Sum consumption, compute common area cost = `roundElectricCharge(commonKwh × blendedRate)`
   b. Call `allocateElectricCommonArea()` to split among active tenants by floor area or equal split
   c. Create `electric_charge` rows with `common_area_share` populated
7. Compute totals: `total_kwh_billed`, `total_billed` (base charges only), `total_admin_fees`, `reconciliation_diff`
8. Create `electric_billing_run` with status=DRAFT
9. Return the run with all charges

**Returns `BAD_REQUEST` if:**
- Any meter reading has negative consumption (current < previous) and is NOT flagged as estimated
- Total base billed > Meralco total_amount (over-recovery — no-markup violation)

#### `electric.finalize`

```ts
input: z.object({
  electricBillingRunId: z.number().int().positive(),
})
```

**Process (transaction-wrapped):**
1. Verify run status = DRAFT (return `PRECONDITION_FAILED` if FINALIZED)
2. Verify total_billed ≤ Meralco bill total_amount (reconciliation re-check)
3. For each electric_charge in the run:
   a. Create a `charge` record:
      - For base electric charges: `charge_type_id = ELECTRIC`, `base_amount = base_charge + common_area_share`, `vat_rate` from setting, `vat_amount` computed
      - For admin fee (if > 0): Create a separate `charge` record with `charge_type_id = ELECTRIC_ADMIN`, `base_amount = admin_fee`, `vat_rate = 0.12` (admin fees are always VATable), `vat_amount = roundVAT(admin_fee × 0.12)`
   b. Update `electric_charge.charge_id` with the electric pass-through charge ID
4. Update run: `status = FINALIZED`, `finalized_at = now()`, `finalized_by = currentUser`
5. Invalidate caches: `electric.listRuns`, `charge.list`, `billingRun.list`, `tenant.getBalance`

#### `electric.deleteRun`

```ts
input: z.object({
```

Returns `PRECONDITION_FAILED` if status = FINALIZED. Cascades delete to `electric_charge` rows (ON DELETE CASCADE).

## 4. Business Logic

### 4.1 Computation Functions (from `@tsvj/computations`)

All electric billing computation is delegated to the `@tsvj/computations` package. The tRPC router bridges DB data → computation input → computation output → DB persistence.

**Blended Rate Electric Billing** (`computeElectricBilling`):

```
blendedRate = roundElectricRate(meralcoTotalBill / meralcoTotalKwh)
               → 4 decimal places, HALF_UP

baseCharge = roundElectricCharge(consumption × blendedRate)
              → 2 decimal places, HALF_UP

adminFee = roundElectricCharge(consumption × adminFeePerKwh)
            → 2 decimal places, HALF_UP (0 if adminFeePerKwh = 0)

subtotal = baseCharge + adminFee

vatAmount = IF vatRate > 0 THEN roundVAT(subtotal × vatRate) ELSE "0.00"

totalCharge = subtotal + vatAmount
```

Rounding: `roundElectricRate` = HALF_UP to 4 decimal places. `roundElectricCharge` = HALF_UP to 2 decimal places. `roundVAT` = HALF_UP to 2 decimal places.

**Key difference from Water (P2):** Water uses per-tier billing where each tenant starts at the lowest bracket. Electric uses a simple blended rate (total ÷ total kWh). This is simpler but the rate changes every month based on the Meralco bill.

**Common Area Allocation** (`allocateElectricCommonArea`):

```
commonAreaCost = roundElectricCharge(commonAreaKwh × blendedRate)
IF allocationMethod = FLOOR_AREA:
  share_i = roundElectricCharge(commonAreaCost × (floorArea_i / totalFloorArea))
IF allocationMethod = EQUAL_SPLIT:
  share_i = roundElectricCharge(commonAreaCost / activeTenantCount)
```

**Note:** Common area electric allocation uses the same `AllocationInput`/`AllocationResult` types as water allocation. The function reuses the allocation pattern from P2 — only the cost computation input differs (blended rate × kWh instead of tiered computation).

### 4.2 Business Rules Enforced in tRPC Router

| Rule | Enforcement | Source |
|------|-------------|--------|
| Blended rate (NOT per-tier) | Computation uses single `blendedRate = totalAmount / totalKwh` | EPIRA RA 9136, ERC Res. 12/2009 |
| No markup (base total ≤ master) | `electric.run` rejects if total base billed > meralco_bill.total_amount | EPIRA RA 9136 Sec. 43 |
| Admin fee = separate line item | Admin fee stored in separate column, creates separate Charge record on finalization | ERC guidance (must not be embedded in per-kWh rate) |
| Admin fee reasonable | Input validation: `adminFeePerKwh ≤ 1.50` (soft cap, configurable) — warning in UI if > 1.00 | Informal ERC reference |
| VAT = configurable | `electric_vat_treatment` AppSetting determines rate; must be configured before billing run | EOPT Act vs BIR RR conflict |
| VAT treatment required | `electric.run` rejects if setting is `"PENDING"` | Accountant decision required |
| Admin fee always VATable | Admin fee charge always has `vat_rate = 0.12` regardless of pass-through VAT setting | Service charge, not pass-through |
| Common area = separate line | Common area charges stored with separate charge_type (COMMON_ELECTRIC) | Standard practice |
| One reading per meter per period | UNIQUE constraint on `(electric_meter_id, billing_period)` | Data integrity |
| DRAFT can be re-run | Existing DRAFT deleted before recomputing; FINALIZED cannot be re-run | Workflow |
| Finalization creates Charges | Only finalized runs create `charge` records visible to P5 billing | Billing integration |
| Shared meter split | Full charge computed first, then multiplied by `share_pct / 100` | Co-tenancy pattern |
| No tenant-type bifurcation | Same blended rate formula for residential and commercial tenants | Electric billing is uniform |

### 4.3 Edge Cases

| Edge Case | Handling |
|-----------|---------|
| **Shared meters (co-tenancy)** | Compute full meter charge, then apply `share_pct` per co-tenant. Each co-tenant gets their own `electric_charge` row. Sum of co-tenant charges = full meter charge (within rounding tolerance). |
| **Defective meter (estimated)** | `is_estimated = true` on reading. Use average of last 3 readings' consumption for this meter as estimated consumption. If < 3 readings exist, use the most recent non-estimated reading's consumption. Flag meter as DEFECTIVE in `electric_meter.status`. |
| **Negative consumption** | `current_reading < previous_reading`. Two scenarios: (1) meter rollover → compute as `(max_reading_value - previous + current)`; (2) data entry error → reject in `electric.run` unless `is_estimated = true`. Admin must confirm rollover via checkbox in reading entry. |
| **Implausible readings** | Consumption > 3× average of last 3 months. Flagged as warning in UI (not blocking). Admin confirms or corrects before running billing. |
| **Vacant units** | Meters with no active `electric_meter_tenant` assignment. Reading entered for loss detection but no `electric_charge` created. Consumption absorbed as landlord cost (appears in under-recovery). |
| **Zero consumption** | Valid — tenant away for the month. Generates `electric_charge` with `base_charge = 0.00`, `admin_fee = 0.00`, `common_area_share` still allocated. Total = common area share + VAT on common area. |
| **Monthly rate fluctuation** | Blended rate changes every month based on Meralco bill. Rate is frozen on the `electric_billing_run` and each `electric_charge` row at computation time. Historical charges always reflect the month's actual rate. |
| **Power outage / partial month** | Meralco bill reflects actual consumption. Lower kWh on master meter → lower consumption per tenant. No special handling needed — blended rate auto-adjusts. |
| **Admin fee cap** | UI shows yellow warning if admin_fee_per_kwh > ₱1.00 and red warning if > ₱1.50. Not a hard block — admin confirms. No published ERC regulation sets a binding cap. |
| **Rounding variance in reconciliation** | Sum of rounded per-tenant base charges may differ from Meralco total by a few pesos due to rounding. Acceptable if `|reconciliation_diff| ≤ total_amount × 0.01` (1% tolerance). The diff is always "under-recovery" — never over. |
| **New tenant mid-month** | First reading = move-in meter reading. No previous reading → consumption = current − move-in reading. |
| **VAT treatment change** | If accountant changes `electric_vat_treatment` mid-year, previously finalized charges retain their frozen `vat_rate`. Only new billing runs use the updated setting. |
