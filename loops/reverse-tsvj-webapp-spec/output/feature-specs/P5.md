# P5 — Monthly Billing Generation

*Feature Spec | Depends on: F0 (Foundation), P1 (Rent Escalation), P2 (Water Billing), P3 (Electric Billing), P4 (Late Payment Penalties) | Implementation order: 3 of 15 (MVP Phase 1)*

---

## Summary

Generate monthly billing statements and VAT Sales Invoices for all active tenants. For each lease, aggregate the current rent (from `RecurringChargePeriod`), finalized utility charges (water, electric), applied penalties, and any other recurring charges into a single billing run. Assign BIR-compliant gapless sequential invoice numbers via the ATP system. Create `Charge` records and `IssuedDocument` records (invoices) that feed into P6 (Payment Tracking), P11 (Rent Roll), P12 (Tax Data), and P13 (Receipt/Invoice Data). Support both batch billing (all active tenants) and individual billing (single tenant). Handle pro-rated first/last months, credit memos for corrections, and prior balance carry-forward.

### 2.4 Procedure Details

#### `billingRun.preview`

Dry-run that computes what charges would be generated without writing to DB.

1. Determine target leases:
   - BATCH: all leases WHERE `status IN ('ACTIVE', 'MONTH_TO_MONTH')` AND `date_start <= last_day_of(billingPeriod)`
   - INDIVIDUAL: single lease for `tenantId`
2. For each lease, aggregate charges:
   - **Rent:** Find active `recurring_charge_period` covering the billing period. Use `amount` from that period. Apply VAT determination (see §4.1).
   - **Water:** Find `water_charge` records WHERE `water_billing_run.billing_period = billingPeriod AND water_billing_run.status = 'FINALIZED'` for this lease's tenants.
   - **Electric:** Same pattern with `electric_charge` + `electric_billing_run`.
   - **Penalties:** Find `penalty_ledger` records WHERE `billing_period = billingPeriod` for this lease.
   - Skip charge types where `include*` is false.
3. Check for warnings (missing TIN, unfinalised utility runs, already billed, etc.)
4. Compute totals per tenant and overall.
5. Return `BillingRunPreview` — no DB writes.

#### `billingRun.create` (transaction-wrapped)

Within `db.transaction()`:

1. **Validate:**
   - No FINALIZED billing run exists for this `billingPeriod` (+ `tenantId` for INDIVIDUAL). DRAFT is OK (will be replaced).
   - If replacing a DRAFT: delete the existing DRAFT billing run and its DRAFT charges.
   - At least one active lease exists for the period.

2. **Create billing run record** in DRAFT status.

3. **For each lease,** generate charge records:
   - **Rent charge:** From active `recurring_charge_period`. If lease started mid-month, pro-rate: `baseAmount = amount × (daysInPeriod / totalDaysInMonth)` using `@tsvj/computations` billing module. Set `source_type = 'RENT'`, `source_id = recurring_charge_period.id`.
   - **Water charges:** Copy from finalized `water_charge` records. Set `source_type = 'WATER'`, `source_id = water_charge.id`. Includes WATER and COMMON_WATER charge types.
   - **Electric charges:** Copy from finalized `electric_charge` records. Set `source_type = 'ELECTRIC'`, `source_id = electric_charge.id`. Includes ELECTRIC, ELECTRIC_ADMIN, and COMMON_ELECTRIC charge types.
   - **Penalty charges:** Copy from `penalty_ledger` records. Set `source_type = 'PENALTY'`, `source_id = penalty_ledger.id`.

4. **For each charge,** apply VAT determination:
   - Call `@tsvj/computations` billing module: `determineVAT({ chargeTypeCode, leaseRegime, currentRent, unitType })` → returns `{ vatRate, isVatExempt }`.
   - Compute: `vatAmount = roundVAT(baseAmount × vatRate)`, `totalAmount = baseAmount + vatAmount`.

5. **Set due date** per charge: `lease.custom_due_day` of the billing month. Default 5 for controlled residential, per-contract for commercial. If `custom_due_day > days_in_month`, clamp to last day.

6. **Insert all charges** in DRAFT status with the billing_run_id set.

7. **Update billing run totals:** `total_base_amount`, `total_vat_amount`, `total_amount`, `charge_count`, `tenant_count`.

8. Return the billing run.

#### `billingRun.finalize` (transaction-wrapped)

The critical mutation that locks charges and generates invoices.

Within `db.transaction()`:

1. **Validate:**
   - Billing run exists and `status = 'DRAFT'`. Return `PRECONDITION_FAILED` if already FINALIZED.
   - Active ATP exists for INVOICE document type. Return `PRECONDITION_FAILED` if no active ATP.
   - ATP has capacity for `tenant_count` more invoices. Return `PRECONDITION_FAILED` if would exceed.

2. **Update all charges** in this billing run to `status = 'FINALIZED'`.

3. **Group charges by tenant** (using `tenant_id`).

4. **For each tenant group,** create an `IssuedDocument`:
   a. **Assign invoice number** atomically via `getNextDocumentNumber(tx, 'INVOICE', propertyId)` (see database-schema §9.1).
   b. **Snapshot tenant data:** `tenant_name_snapshot`, `tenant_tin_snapshot`, `tenant_address_snapshot` — frozen at this moment.
   c. **Compute VAT summary:** `vatable_sales` (sum of base_amount WHERE vat_rate > 0), `vat_exempt_sales` (sum of base_amount WHERE is_vat_exempt = true), `zero_rated_sales = 0`.
   d. **Insert `IssuedDocument`** with `document_type = 'INVOICE'`.
   e. **Insert `IssuedDocumentLine`** records — one per charge, ordered by charge type (RENT first, then utilities, then penalties, then other).
   f. **Check ATP utilization** after each number assignment. If >= 80%: create ATP_EXHAUSTION alert.

5. **Update billing run:** `status = 'FINALIZED'`, `finalized_at = now()`, `finalized_by = userId`.

6. **Refresh materialized view:** `REFRESH MATERIALIZED VIEW CONCURRENTLY tenant_balance`.

7. **Check for Form 2307 tracking** (for corporate tenants): for each corporate tenant billed, check if a `form_2307_record` exists for the current quarter. If not, create a PENDING record with `expected_amount` = sum of `base_amount` for rent charges × 0.05. Create `FORM_2307_MISSING` alert if not already created for this quarter.

8. Return `BillingRunFinalizeResult` with invoice numbers and ATP utilization.

#### `billingRun.delete`

1. Validate: billing run exists and `status = 'DRAFT'`. Return `PRECONDITION_FAILED` if FINALIZED.
2. Delete all DRAFT charges with this `billing_run_id`.
3. Delete the billing run.

#### `billingRun.createCreditMemo` (transaction-wrapped)

Cannot void a BIR invoice — must issue credit memo (RR 16-2005 Sec. 4.113-5).

Within `db.transaction()`:

1. **Validate:**
   - Original charge exists and `status = 'FINALIZED'` (not DRAFT, CREDITED, or PAID).
   - Credit amount ≤ original charge `total_amount` (for partial credits, `base_amount` portion ≤ original `base_amount`).
   - Active ATP exists for CREDIT_MEMO document type.

2. **Create credit charge:** A new `charge` record with:
   - Negative `base_amount` (= `-creditMemoInput.amount`)
   - `vat_rate` matching the original charge
   - `vat_amount` = `-roundVAT(creditMemoInput.amount × original.vatRate)`
   - `status = 'FINALIZED'` (credit memos are finalized immediately)
   - `charge_type_id` = original charge's type
   - `source_type = 'CREDIT_MEMO'`

3. **Update original charge status** to `CREDITED` if full credit (amount = original base_amount), otherwise leave as FINALIZED (partial credit tracked via `credit_memo` link).

4. **Assign credit memo document number** via `getNextDocumentNumber(tx, 'CREDIT_MEMO', propertyId)`.

5. **Create CreditMemo record** linking original charge to credit charge.

6. **Create IssuedDocument** with `document_type = 'CREDIT_MEMO'` + lines.

7. **Refresh materialized view.**

8. Return the credit memo.

#### `billingRun.getCharges`

Paginated list of charges with filtering by tenant, lease, period, status, charge type, and billing run. Includes related tenant name and charge type name.

#### `billingRun.getChargeSummary`

Per-tenant charge summary for a billing period. Aggregates charges + includes prior balance from `tenant_balance` materialized view. Used by the billing statement PDF.

## 4. Business Logic

### 4.1 VAT Determination (from `@tsvj/computations` billing module)

The billing module centralizes VAT determination logic, called per charge during billing run creation:

```
determineVAT(input):
  IF chargeTypeCode = 'WATER' or 'COMMON_WATER':
    → { vatRate: '0.00', isVatExempt: false }  // Conduit, not exempt — BIR RR 16-2005

  IF chargeTypeCode = 'ELECTRIC' or 'ELECTRIC_ADMIN' or 'COMMON_ELECTRIC':
    IF electricVatTreatment = 'PENDING':
      → ERROR: "Electric VAT treatment not configured"
    IF electricVatTreatment = 'EXEMPT':
      → { vatRate: '0.00', isVatExempt: false }
    IF electricVatTreatment = 'VATABLE':
      → { vatRate: '0.12', isVatExempt: false }

  IF unitType = 'RESIDENTIAL' AND currentMonthlyRent ≤ 15000:
    → { vatRate: '0.00', isVatExempt: true }    // NIRC Sec. 109(1)(Q)

  IF lessorAggregateAnnualReceipts ≤ 3000000:
    → { vatRate: '0.00', isVatExempt: false }    // Below threshold

  → { vatRate: '0.12', isVatExempt: false }      // Standard 12% VAT
```

**Note:** The lessor aggregate annual receipts check uses `AppSettings.is_vat_registered` as a proxy (if the corporation is VAT-registered, it exceeds the threshold). This simplifies the check to a boolean flag rather than computing aggregate receipts.

### 4.2 Charge Builder (from `@tsvj/computations` billing module)

```ts
function buildCharge(input: {
  baseAmount: string;
  vatDetermination: { vatRate: string; isVatExempt: boolean };
}): { baseAmount: string; vatRate: string; vatAmount: string; totalAmount: string; isVatExempt: boolean } {
  const base = peso(input.baseAmount);
  const rate = peso(input.vatDetermination.vatRate);
  const vat = roundVAT(base.mul(rate));  // HALF_UP to 2 decimals
  const total = base.add(vat);
  return {
    baseAmount: base.toString(),
    vatRate: rate.toString(),
    vatAmount: vat.toString(),
    totalAmount: total.toString(),
    isVatExempt: input.vatDetermination.isVatExempt,
  };
}
```

### 4.3 Pro-Ration (from `@tsvj/computations` billing module)

For first/last month billing when lease starts/ends mid-month:

```ts
function proRateCharge(input: {
  fullMonthAmount: string;
  startDate: string;         // First day tenant is liable
  endDate: string;           // Last day tenant is liable
  billingPeriodStart: string; // First day of billing month
  billingPeriodEnd: string;   // Last day of billing month
}): { proRatedAmount: string; days: number; totalDays: number } {
  const totalDays = daysBetween(input.billingPeriodStart, input.billingPeriodEnd) + 1;
  const start = max(input.startDate, input.billingPeriodStart);
  const end = min(input.endDate, input.billingPeriodEnd);
  const days = daysBetween(start, end) + 1;
  const proRated = roundMoney(peso(input.fullMonthAmount).mul(days).div(totalDays));
  return { proRatedAmount: proRated.toString(), days, totalDays };
}
```

Pro-ration applies when:
- Lease `date_start` is after the 1st of the billing month (first month)
- Lease `date_end` is before the last day of the billing month (last month)

### 4.4 Due Date Calculation

```
dueDate = billingPeriod(year, month) + lease.custom_due_day
IF custom_due_day > daysInMonth(year, month):
  dueDate = lastDayOfMonth(year, month)

Default custom_due_day:
  CONTROLLED_RESIDENTIAL: 5  (RA 9653 Sec. 7)
  NON_CONTROLLED_RESIDENTIAL: per contract (default 5)
  COMMERCIAL: per contract (default 15)
```

### 4.5 Charge Aggregation Order

During billing run creation, charges are generated in this order per lease:
1. **RENT** — from `recurring_charge_period` (always first)
2. **WATER** + **COMMON_WATER** — from `water_charge`
3. **ELECTRIC** + **ELECTRIC_ADMIN** + **COMMON_ELECTRIC** — from `electric_charge`
4. **PENALTY** — from `penalty_ledger`
5. **OTHER** — future charge types (not in MVP)

This order is preserved in `issued_document_line.line_number`.

### 4.6 Invoice Generation Rules (RR 7-2024)

- Invoice issued at **accrual** (when charges become due), not at payment.
- Output VAT is recognized at invoice issuance date.
- All 16 mandatory fields per RR 7-2024 must be populated (see document-generation §4.3).
- Missing tenant TIN → warning but not blocking (tenant can be invoiced without TIN, but the invoice is deficient per BIR rules).
- ATP exhaustion → blocking (cannot issue invoice without valid ATP range).

### 4.7 Credit Memo Rules (RR 16-2005 Sec. 4.113-5)

- A BIR invoice **cannot be voided or deleted**. To correct, issue a credit memo.
- Credit memo has its own gapless sequential series (separate from invoices and receipts).
- Credit memo references the original invoice number.
- The credit amount reduces output VAT in the period the credit memo is issued.
- The original charge's status changes to CREDITED (if full credit) or remains FINALIZED (if partial credit with balance remaining).
