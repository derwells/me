# P14 — Expense Tracking

*Feature Spec | Depends on: F0 (Foundation — property, tenant, charge_type, app_settings, alert, audit_log, auth stack, app shell) | Implementation order: 13 of 15 (MVP Wave 4: P7+P12+P13+P14)*

---

## Summary

Record all property disbursements (repairs, maintenance, utilities, permits, supplies, professional fees) with automatic EWT withholding computation, input VAT tracking, and optional capital expenditure classification with straight-line depreciation schedules. Compile into structured exports (expense register, input VAT register, EWT summary, Form 2307 for suppliers, QAP data, fixed asset schedule) that the external accountant uses for BIR filings.

### 1.5 Voucher Number Sequence

Voucher numbers follow the pattern `DV-YYYYMM-NNNN` (e.g., `DV-202603-0001`). Generated atomically on voucher creation:

```ts
// In expense.create mutation
const currentMonth = format(input.dateVoucher, "yyyyMM");
const lastVoucher = await db.query.disbursementVoucher.findFirst({
  where: sql`voucher_number LIKE ${"DV-" + currentMonth + "-%"}`,
  orderBy: desc(disbursementVoucher.voucherNumber),
});
const nextSeq = lastVoucher
  ? parseInt(lastVoucher.voucherNumber.split("-")[2]) + 1
  : 1;
const voucherNumber = `DV-${currentMonth}-${String(nextSeq).padStart(4, "0")}`;
```

#### Voucher Management

**expense.list**
- **Type:** query

**expense.getById**
- **Type:** query
- **Auth:** protectedProcedure
- **Input:** `{ id: number }`
- **Output:** Full voucher with joined supplier, category, input_vat_record, ewt_withheld_record, fixed_asset (if capital), audit trail
- **Logic:** Single voucher detail with all related records.

**expense.create**
- **Type:** mutation
- **Auth:** adminProcedure
- **Input:**
```ts
z.object({
  propertyId: z.number(),
  supplierId: z.number(),
  expenseCategoryId: z.number(),
  dateVoucher: z.string().date(),
  description: z.string().min(3).max(500),
  grossAmount: z.string().regex(/^\d+\.\d{2}$/),
  supportingDocRef: z.string().optional(),
  paymentMethod: z.enum(["CHECK", "BANK_TRANSFER", "CASH"]).optional(),
  checkNumber: z.string().optional(),
  bankAccount: z.string().optional(),
  isCapitalExpenditure: z.boolean().default(false),
  // Capital expenditure fields (required if isCapitalExpenditure)
  usefulLifeMonths: z.number().int().positive().optional(),
  assetDescription: z.string().optional(),
}).refine(
  (d) => !d.isCapitalExpenditure || (d.usefulLifeMonths && d.assetDescription),
  { message: "Capital expenditure requires useful life and asset description" }
)
```
- **Output:** Created voucher with computed fields
- **Logic (transaction-wrapped):**
  1. Look up supplier_payee → get payee_type, is_vat_registered, is_government
  2. Look up expense_category → get ewt_rate_individual, ewt_rate_corporate, is_ewt_applicable
  3. Compute EWT via `@tsvj/computations` `computeEWTOnSupplier()`:
     - If supplier.is_government OR !category.is_ewt_applicable → ewt_rate=0, ewt_amount=0
     - Else → rate from category by payee_type, ewt_amount = roundEWT(grossAmount × rate)
  4. Compute input VAT:
     - If supplier.is_vat_registered → vat_amount = roundVAT(grossAmount × 0.12)
     - Else → vat_amount = 0
  5. Compute net_payment = grossAmount + vat_amount − ewt_amount
  6. Generate voucher_number (DV-YYYYMM-NNNN)
  7. INSERT disbursement_voucher as DRAFT
  8. If vat_amount > 0: INSERT input_vat_record
     - Determine quarter from dateVoucher
     - is_creditable = supplier.is_vat_registered AND !supplier.is_government
     - apportionment_pct from AppSettings (default 1.0 if not mixed-operation)
     - creditable_amount = roundVAT(vat_amount × apportionment_pct)
  9. If ewt_amount > 0: INSERT ewt_withheld_record
     - period_month = format(dateVoucher, "yyyy-MM")
     - period_quarter = computed from dateVoucher
     - Snapshot supplier TIN and name
  10. If isCapitalExpenditure:
      - residual_value = roundMoney(grossAmount × 0.05) — 5% floor
      - INSERT fixed_asset_register
      - Generate depreciation_schedule_entry records (monthly SLM from date_acquired through end of useful life)
  11. INSERT audit_log

**expense.update**
- **Type:** mutation
- **Auth:** adminProcedure
- **Input:** `{ id: number, ...partial voucher fields }`
- **Output:** Updated voucher
- **Logic:** Only DRAFT vouchers can be updated. Recalculates EWT/VAT/net if gross_amount, supplier, or category changed. Updates related input_vat_record and ewt_withheld_record. Regenerates depreciation schedule if capital fields changed.

**expense.approve**
- **Type:** mutation
- **Auth:** adminProcedure
- **Input:** `{ id: number }`
- **Output:** Updated voucher
- **Logic:** Validates status = DRAFT → set status = APPROVED, approved_by = ctx.userId, approved_at = now(). Returns PRECONDITION_FAILED if not DRAFT.

**expense.markPaid**
- **Type:** mutation
- **Auth:** adminProcedure
- **Input:** `{ id: number, datePaid: z.string().date(), paymentMethod: z.enum(["CHECK", "BANK_TRANSFER", "CASH"]), checkNumber?: string, bankAccount?: string }`
- **Output:** Updated voucher
- **Logic:** Validates status = APPROVED → set status = PAID, date_paid = datePaid, payment details. Returns PRECONDITION_FAILED if not APPROVED. INSERT audit_log.

**expense.void**
- **Type:** mutation
- **Auth:** adminProcedure
- **Input:** `{ id: number, reason: z.string().min(5) }`
- **Output:** Updated voucher
- **Logic:** Validates status IN (DRAFT, APPROVED) — cannot void PAID vouchers. Set status = VOID. Mark related input_vat_record as is_creditable = false. Mark ewt_withheld_record with voided = true (or delete). INSERT audit_log with reason.

## 4. Business Logic

### 4.1 EWT Computation — `@tsvj/computations` tax module

Uses the existing `computeEWTOnSupplier()` function from `src/tax/ewt-supplier.ts`:

```ts
// Input: supplier profile + expense category + gross amount
// Output: EWT rate, amount, input VAT, net payment, ATC code

computeEWTOnSupplier({
  grossAmount: "15000.00",
  payeeType: "CORPORATE",
  isVATRegistered: true,
  isLargeTaxpayer: false,
  ewtRateIndividual: "0.02",  // From expense_category
  ewtRateCorporate: "0.02",   // From expense_category
})
// → { ewtRate: "0.02", ewtAmount: "300.00", inputVATCreditable: true,
//    inputVATAmount: "1800.00", netPayment: "16500.00", atcCode: "WC120" }
```

**EWT rate determination rules:**
- Government payees: always 0% (no EWT)
- Direct utility payments (Meralco, Maynilad): 0% (utility cos under final withholding)
- Individual payees: use ewt_rate_individual from expense_category
- Corporate payees: use ewt_rate_corporate from expense_category
- EWT base = gross_amount (excluding VAT — VAT is NOT withheld)
- EWT amount = roundEWT(gross_amount × rate) — HALF_UP to 2 decimals
- Professional fees: rate depends on gross income bracket (sworn declaration)
  - Default to higher rate if no declaration on file

### 4.2 Input VAT Determination

```ts
// From computeEWTOnSupplier result
inputVATCreditable: boolean  // true only if supplier.is_vat_registered
inputVATAmount: string       // roundVAT(grossAmount × 0.12) if creditable, else "0.00"
```

**Mixed-operation apportionment** (if TSVJ has both VATable and exempt income):
- Directly attributable expenses: full credit (or zero) based on attributable income stream
- Common expenses: creditable_amount = vat_amount × (VATable revenue / total revenue)
- Apportionment percentage stored in AppSettings, computed quarterly

**Post-2022 capital goods:** All input VAT from capital goods is recognized immediately (no 60-month amortization). The `is_capital_good` flag on input_vat_record is for 2550Q line item reporting only.

### 4.3 Depreciation Computation — `@tsvj/computations` tax module

New function in `src/tax/depreciation.ts`:

```ts
export interface DepreciationInput {
  cost: string;              // Acquisition cost
  residualValue: string;     // ≥ 5% of cost
  usefulLifeMonths: number;  // Taxpayer judgment
  startDate: string;         // Date acquired (YYYY-MM-DD)
}

export interface DepreciationSchedule {
  monthlyAmount: string;     // (cost − residualValue) / usefulLifeMonths
  entries: DepreciationEntry[];
}

export interface DepreciationEntry {
  period: string;            // "YYYY-MM"
  amount: string;            // Monthly depreciation
  accumulatedTotal: string;  // Running total
}

export function computeDepreciationSchedule(input: DepreciationInput): DepreciationSchedule;
```

**Rules:**
- Straight-line method: monthly = (cost − residual) / useful_life_months
- Residual value ≥ 5% of cost (RR 19-86 Sec. 4.03)
- Monthly amount rounded to 2 decimals (HALF_UP)
- Last month absorbs rounding remainder: depreciable_base − sum_of_prior_entries
- Schedule starts from month of acquisition through useful_life_months
- First month: full month (no pro-ration for partial months — simplification for MVP)

### 4.4 Voucher Number Generation

Sequential, gapless within each month. Pattern: `DV-YYYYMM-NNNN`.

Generated atomically in the `expense.create` mutation using a SELECT MAX + 1 pattern within the transaction. VOID vouchers consume their number (gapless = no reuse).

### 4.5 Form 2307 Generation

Form 2307 (Certificate of Creditable Tax Withheld at Source) is issued to each supplier quarterly:

- Generated per payee per quarter from aggregated ewt_withheld_record rows
- Payor info from AppSettings (company TIN, name, address, RDO)
- Payee info from supplier_payee (TIN, name, address)
- Income payment details: ATC code, gross amount, EWT rate, EWT withheld
- Period: quarter covered
- PDF rendered via @react-pdf/renderer at `/api/pdf/form2307/[supplierId]?quarter=2026-Q1`
- Batch download: ZIP of all Form 2307 PDFs for a quarter

## 7. Edge Cases

1. **Government payee (RPT, permits):** EWT = 0%, input VAT = 0%. System skips ewt_withheld_record and input_vat_record creation. No Form 2307 generated.

2. **Utility payments (Meralco, Maynilad):** EWT = 0% (utility companies under final withholding). Input VAT = creditable (utilities are VAT-registered). ewt_withheld_record not created but input_vat_record IS created.

3. **Non-VAT supplier:** EWT computed normally from category rate. Input VAT = 0 (nothing to credit). No input_vat_record created.

4. **Professional fee rate ambiguity:** If no sworn declaration on file, system defaults to higher rate bracket (10% for individual, 15% for corporate). Admin can override by selecting specific category (WI010 vs WI011, WC010 vs WC011).

5. **Capital vs. ordinary classification:** User selects checkbox. System enforces: if capital, useful life and description required. No automatic classification — this requires human judgment per process catalog.

6. **Voided voucher:** VOID only from DRAFT or APPROVED. Cannot void PAID (requires reversal workflow out of MVP scope). VOID sets input_vat_record.is_creditable = false. Voided voucher numbers are consumed (gapless).

7. **Mid-quarter supplier change:** If supplier's payee_type or VAT registration changes mid-quarter, existing vouchers are NOT retroactively updated. New vouchers use the current profile. Form 2307 aggregates by supplier regardless of profile changes.

8. **Mixed-operation apportionment:** If TSVJ has both VATable (commercial) and exempt (residential ≤₱15K) tenants, input VAT on common expenses is apportioned. The apportionment_pct is configured in AppSettings and updated quarterly by the admin based on revenue mix.

9. **Depreciation of disposed asset:** When an asset is disposed mid-period, depreciation stops at the disposal month. The disposed asset remains in the register with DISPOSED status. Any gain/loss on disposal (proceeds vs NBV) is tracked for income tax purposes.

10. **Form 2307 for multiple ATC codes:** If a single supplier has vouchers across multiple expense categories (e.g., both WC120 repairs and WC158 services), the Form 2307 PDF groups income payments by ATC code within the same certificate.
