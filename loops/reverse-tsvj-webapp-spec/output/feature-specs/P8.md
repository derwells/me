# P8 — Lease Contract Generation

*Feature Spec | Depends on: F0 (Foundation), P1 (Rent Escalation — for RecurringChargePeriod data used in per-year DST) | Implementation order: 10 of 15 (MVP Wave 3: P8+P9+P10)*

---

## Summary

Generate lease contracts from regime-aware templates with variable substitution, mandatory PH clause assembly (RA 9653 for controlled residential, Civil Code for commercial), DST computation per NIRC Sec. 194 (TRAIN Law), and execution milestone tracking (draft → signed → notarized → registered → DST filed). Produce downloadable PDF contracts via `@react-pdf/renderer` with clause ordering and variable interpolation.

```ts
  mandatoryClauses: z.array(clausePreviewSchema),   // Always included
  optionalClauses: z.array(clausePreviewSchema),     // Selected by user
  assembledClauses: z.array(z.object({               // Final ordered list
    id: z.number(),
    code: z.string(),
    title: z.string(),
    renderedContent: z.string(),                     // Variables substituted
    isMandatory: z.boolean(),
    sortOrder: z.number(),
  })),
  variableMap: z.record(z.string(), z.string()),     // All substituted variables
  dst: dstResultSchema,                               // DST computation preview
  warnings: z.array(z.string()),                      // Validation warnings
  boardResolution: boardResolutionSchema.nullable(),   // Corporate tenant's resolution
})
```

**Logic:**
1. Load lease with tenant, rentables, recurring charge periods
2. Load template and clauses (mandatory for regime + selected optional)
3. Build variable substitution map (see Section 4.2)
4. Substitute variables in each clause's content
5. Compute DST via `@tsvj/computations` `computeDST`
6. Validate: controlled RA 9653 compliance, board resolution coverage (corporate), TIN presence
7. Return assembled preview with warnings

**Warnings generated:**
- `MISSING_TIN`: Corporate tenant has no TIN
- `NO_BOARD_RESOLUTION`: Corporate tenant, no in-force resolution covers this lease
- `RESOLUTION_SCOPE_EXCEEDED`: Lease exceeds resolution's max_monthly_rent or max_lease_years
- `DEPOSIT_EXCEEDS_CAP`: Controlled residential deposit > 2 months
- `ADVANCE_EXCEEDS_CAP`: Controlled residential advance > 1 month
- `ESCALATION_EXCEEDS_NHSB`: Controlled residential escalation rate exceeds NHSB cap
- `MISSING_FLOOR_AREA`: Rentable has no floor_area_sqm recorded

## 3. Business Logic

### 3.1 DST Computation — `@tsvj/computations` `src/contract/dst.ts`

**NIRC Sec. 194 (TRAIN Law, RA 10963, effective Jan 1, 2018):**

| Portion of Annual Rent | DST |
|---|---|
| First PHP 2,000 (or fractional part) | PHP 6.00 |
| Every PHP 1,000 (or fractional part) in excess of PHP 2,000 | PHP 2.00 |

Applied per year of lease term. Partial years rounded up (NIRC "or fractional part thereof").

**Formula:**
```
annualRent = monthlyRent × 12
totalRent = annualRent × leaseTermYears
if totalRent ≤ 2000:
  dstAmount = 6
else:
  excess = totalRent - 2000
  units = CEIL(excess / 1000)
  dstAmount = 6 + (units × 2)
```

**Test vectors:**

| Monthly Rent | Term (years) | Annual Rent | Total Rent | DST |
|---|---|---|---|---|
| PHP 7,200 | 1 | PHP 86,400 | PHP 86,400 | PHP 6 + CEIL(84,400/1000)×2 = PHP 6 + 85×2 = **PHP 176** |
| PHP 15,000 | 2 | PHP 180,000 | PHP 360,000 | PHP 6 + CEIL(358,000/1000)×2 = PHP 6 + 358×2 = **PHP 722** |
| PHP 100 | 1 | PHP 1,200 | PHP 1,200 | **PHP 6** (≤ PHP 2,000) |
| PHP 25,000 | 3 | PHP 300,000 | PHP 900,000 | PHP 6 + CEIL(898,000/1000)×2 = PHP 6 + 898×2 = **PHP 1,802** |
| PHP 2,500 | 1 | PHP 30,000 | PHP 30,000 | PHP 6 + CEIL(28,000/1000)×2 = PHP 6 + 28×2 = **PHP 62** |

**Edge cases:**
- Annual rent exactly PHP 2,000: DST = PHP 6 (only base, no excess)
- Annual rent PHP 2,001: DST = PHP 6 + 1×2 = PHP 8 (fractional PHP 1,000 rounds up)
- Partial-year lease (e.g., 18 months): `leaseTermYears = 2` (CEIL(18/12) — fractional year rounds up per NIRC)

### 3.2 Board Resolution Scope Validation

```
checkResolutionCovers(resolution, lease):
  if not resolution.is_in_force:
    return { valid: false, reason: "RESOLUTION_NOT_IN_FORCE" }
  if resolution.max_monthly_rent is not null AND lease.monthly_rate > resolution.max_monthly_rent:
    return { valid: false, reason: "EXCEEDS_RENT_LIMIT" }
  if resolution.max_lease_years is not null AND lease.term_years > resolution.max_lease_years:
    return { valid: false, reason: "EXCEEDS_TERM_LIMIT" }
  return { valid: true }
```

This validation produces warnings (not errors) — corporate tenants may have implied authority via officers (President/CEO). The admin makes the final judgment call.

### 3.3 Controlled Residential Validation

At contract generation, validate RA 9653 constraints. These are warnings (admin may override if vacancy decontrol applies):

1. **Deposit cap:** `deposit_amount ≤ 2 × monthly_rent` (RA 9653 Sec. 7)
2. **Advance cap:** `advance_rent ≤ 1 × monthly_rent` (RA 9653 Sec. 7)
3. **Escalation cap:** `escalation_rate ≤ NHSB cap for year` (RA 9653 Sec. 4)
4. **Grace period:** Must be at least 5 days (RA 9653 Sec. 7)
5. **Penalty cap:** Must not exceed 1%/month (Art. 1229 safe harbour)

These validations reuse the lease form's `.superRefine()` logic (F0). At contract generation, they are advisory — the lease data was already validated at creation.

### 3.5 DST Filing Deadline Computation

```
filingDeadline(executionDate):
  monthEnd = last_day_of_month(executionDate)
  return monthEnd + 5 calendar days
```

Example: Contract signed March 15, 2026 → Month end March 31, 2026 → Filing deadline April 5, 2026.

## 7. Edge Cases

1. **Corporate tenant without board resolution:** Warning generated at preview, but generation is not blocked. Admin may proceed if officer has implied authority (President/CEO per jurisprudence). The system tracks the absence for audit purposes.

2. **Multi-rentable lease:** One lease covering multiple units (e.g., Unit 101A + 101B). Property description clause concatenates all rentable names and floor areas. DST computed on aggregate monthly rent.

3. **Partial-year lease term:** A lease of 18 months → `leaseTermYears = 2` for DST purposes (NIRC "or fractional part thereof" rounds up). The per-year breakdown shows only full years.

4. **Vacancy decontrol:** New tenant in previously controlled unit with rent set above PHP 10,000 → regime is COMMERCIAL or NON_CONTROLLED_RESIDENTIAL. Template selection follows the lease's regime, not the unit's historical regime.

5. **Standing board resolution:** Covers multiple leases without per-lease approval. `is_standing = true` + scope limits (max_monthly_rent, max_lease_years). System validates lease falls within scope.

6. **Per-year escalated rent DST:** If recurring charge periods show different monthly rents across years (e.g., Year 1: ₱10,000, Year 2: ₱12,000), DST should ideally be computed per year. The current `computeDST` function uses a single monthly rent — for stepped escalation, the caller should use the initial monthly rent (conservative approach matching common practice). Future enhancement could support per-year breakdown.

7. **Template/clause versioning:** When a template or clause is updated after a contract was generated, the PDF re-renders with current clause text. The `lease_event.details` stores the clause IDs used at generation time, but not the clause text snapshot. For immutability, a future enhancement could snapshot clause content at generation. For MVP, this is acceptable since templates rarely change and contracts are downloaded promptly after generation.

8. **DST filed before signing:** The system computes `filing_deadline` only when the SIGNED milestone is recorded (since the 5-day rule runs from month of execution). If DST is filed before signing (unusual but possible), admin can record via `updateDSTFiling` independently.

9. **Lease already has contract:** Prevent duplicate generation — if a DRAFT milestone exists, the lease is excluded from the wizard's lease selector. To regenerate, admin must first delete the existing DST computation (not implemented in MVP — edge case for future enhancement).

10. **Missing variables:** If a variable placeholder in a clause has no matching data (e.g., `{{tenant_tin}}` when TIN is null), replace with `[___________]` and add to warnings list. Admin can fill in manually on the printed copy.
