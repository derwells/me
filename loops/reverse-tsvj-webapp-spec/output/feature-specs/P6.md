# P6 — Tenant Payment Tracking

*Feature Spec | Depends on: F0 (Foundation), P4 (Late Payment Penalties), P5 (Monthly Billing Generation) | Implementation order: 4 of 15 (MVP Phase 1)*

---

## Summary

Record tenant payments, allocate against outstanding charges following the Civil Code Art. 1252-1254 hierarchy, generate Official Receipts with BIR-mandated gapless numbering, handle EWT withholding for corporate tenants, maintain running balances per tenant via the TenantBalance materialized view, track Form 2307 certificates, and provide arrears monitoring with dashboard visibility. This feature closes the billing-to-collection loop opened by P5.

### 1.2 TenantBalance Materialized View

Already defined in the database-schema analysis (§8). P6 is the feature that creates and first uses it. The view definition:

```sql
CREATE MATERIALIZED VIEW tenant_balance AS
SELECT
  t.id AS tenant_id,
  COALESCE(SUM(c.total_amount) FILTER (
    WHERE c.status IN ('FINALIZED', 'PAID', 'PARTIALLY_PAID')
  ), 0) AS total_billed,
  COALESCE(SUM(pa.amount), 0) AS total_paid,
  COALESCE(SUM(p.ewt_withheld) FILTER (WHERE p.is_reversed = false), 0) AS total_ewt,
  COALESCE(SUM(c.total_amount) FILTER (
    WHERE c.status IN ('FINALIZED', 'PAID', 'PARTIALLY_PAID')
  ), 0)
    - COALESCE(SUM(pa.amount), 0)
    - COALESCE(SUM(p.ewt_withheld) FILTER (WHERE p.is_reversed = false), 0) AS balance,
  MIN(c.due_date) FILTER (
    WHERE c.status IN ('FINALIZED', 'PARTIALLY_PAID')
  ) AS oldest_unpaid_date,
  COUNT(DISTINCT c.billing_period) FILTER (
    WHERE c.status IN ('FINALIZED', 'PARTIALLY_PAID')
  ) AS months_in_arrears,
  MAX(p.date_issued) FILTER (WHERE p.is_reversed = false) AS last_payment_date,
  NOW() AS last_refreshed
FROM tenant t
LEFT JOIN charge c ON c.tenant_id = t.id
LEFT JOIN payment_allocation pa ON pa.charge_id = c.id
LEFT JOIN payment p ON p.tenant_id = t.id AND p.is_reversed = false
GROUP BY t.id
WITH DATA;

CREATE UNIQUE INDEX idx_tenant_balance_tenant_id ON tenant_balance (tenant_id);
```

**Refresh triggers:** Called via `REFRESH MATERIALIZED VIEW CONCURRENTLY tenant_balance` inside the same transaction after:
1. `payment.create` (new payment recorded)
2. `payment.reverse` (payment reversed)
3. `billingRun.finalize` (new charges created) — already in P5
4. `billingRun.createCreditMemo` (charge credited) — already in P5

## 3. Business Logic

### 3.1 Payment Recording Flow

```
FUNCTION payment.create(input):
  TRANSACTION {
    1. Validate tenant exists, is on an ACTIVE/MONTH_TO_MONTH lease
    2. Validate amount > 0, amount + ewtWithheld > 0

    3. Fetch outstanding charges for tenant:
       SELECT * FROM charge
       WHERE tenant_id = input.tenantId
         AND status IN ('FINALIZED', 'PARTIALLY_PAID')
       ORDER BY due_date ASC

    4. Compute allocation via @tsvj/computations allocatePayment():
       - If input.designation provided (Art. 1252):
         a. Validate all designated charge IDs belong to this tenant
         b. Validate all designated charges are FINALIZED or PARTIALLY_PAID
         c. Art. 1253 STILL applies: if designated charges include principal
            alongside penalties on that same debt, penalties are satisfied first
         d. Allocate to designated charges in order provided
       - If no designation (automatic allocation):
         a. Sort: penalty/interest charges FIRST (Art. 1253 — mandatory)
         b. Within same priority: oldest due_date first (Art. 1254 FIFO)
         c. Allocate until payment exhausted or all charges satisfied

    5. EWT reconciliation (corporate tenants):
       - effectivePayment = cashReceived + ewtWithheld
       - For each allocated charge:
         ewtPortion = chargeBaseAmount × ewtRate (5% rent/penalty, 2% utility)
         cashPortion = allocatedAmount - ewtPortion
       - Validate: total allocated ≤ effectivePayment

    6. Insert payment record

    7. Insert payment_allocation records (one per charge touched)

    8. Update charge statuses:
       FOR EACH allocation:
         totalPaidOnCharge = SUM(all allocations for this charge)
         IF totalPaidOnCharge >= charge.total_amount:
           UPDATE charge SET status = 'PAID'
         ELSE:
           UPDATE charge SET status = 'PARTIALLY_PAID'

    9. Generate Official Receipt (if input.generateReceipt = true):
       a. Call getNextDocumentNumber(tx, 'RECEIPT', propertyId)
       b. Insert issued_document (type=RECEIPT, linked to payment)
       c. Insert issued_document_line records (one per allocation)
       d. Check ATP utilization → create alert if ≥80%/90%/100%
       e. Update payment.or_number and payment.or_document_id

    10. Form 2307 tracking (corporate tenants, ewtWithheld > 0):
        a. Determine quarter from input.dateIssued
        b. UPSERT form_2307_record:
           - If exists: ADD ewtWithheld to expected_amount
           - If not: CREATE with expected_amount = ewtWithheld

    11. Arrears check:
        a. Count remaining unpaid billing periods for this tenant
        b. If tenant is on CONTROLLED_RESIDENTIAL lease and months_in_arrears >= 3:
           - Create ARREARS alert (level=OVERDUE, category=ARREARS)
        c. If months_in_arrears >= 2 (controlled):
           - Create ARREARS alert (level=WARNING)

    12. Refresh materialized view:
        REFRESH MATERIALIZED VIEW CONCURRENTLY tenant_balance

    13. Log payment event:
        INSERT payment_event (type=CREATED, details={allocations, ewtWithheld})

    14. Return payment + allocations + orDocument
  }
```

### 3.2 Payment Reversal Flow

```
FUNCTION payment.reverse(input):
  TRANSACTION {
    1. Fetch payment, validate:
       - Payment exists
       - Payment is NOT already reversed (is_reversed = false)
       - Payment was created by a user (not a system operation)

    2. Restore charge statuses:
       FOR EACH allocation on this payment:
         a. DELETE the payment_allocation
         b. Re-compute remaining balance on charge:
            remainingAllocations = SUM(all other allocations for charge)
            IF remainingAllocations = 0:
              UPDATE charge SET status = 'FINALIZED'
            ELSE IF remainingAllocations < charge.total_amount:
              UPDATE charge SET status = 'PARTIALLY_PAID'

    3. Mark payment as reversed:
       UPDATE payment SET
         is_reversed = true,
         reversed_at = NOW(),
         reversed_by = ctx.userId,
         reversal_reason = input.reason

    4. Handle OR: The issued receipt document is NOT deleted (BIR rules:
       document numbers are never voided). Instead, mark the issued_document
       with a note "REVERSED" in metadata. A new reversed receipt document
       is not created — the reversal is tracked via payment_event.

    5. Form 2307 adjustment (if corporate):
       a. If payment had ewt_withheld > 0:
          - SUBTRACT from form_2307_record.expected_amount for that quarter
          - If expected_amount becomes 0, mark as reconciled

    6. Re-check arrears (may worsen after reversal)

    7. Refresh materialized view

    8. Log payment event (type=REVERSED, details={reason, original allocations})
  }
```

### 3.3 EWT Reconciliation Formula

For corporate tenants, the payment equation per charge is:

```
invoice_total = base_amount + vat_amount
cash_received + ewt_withheld = invoice_total
WHERE ewt_withheld = base_amount × ewt_rate

EWT rates by charge type:
  RENT        → 5% (ATC: WC157)
  PENALTY     → 5% (ATC: WC157, treated as additional rental)
  WATER       → 2% (ATC: WC160, reimbursement)
  ELECTRIC    → 2% (ATC: WC160, reimbursement)
  OTHER       → 5% (ATC: WC157, default)
```

The `@tsvj/computations` `computeEWTOnRent()` function determines the correct rate per charge type. The tRPC router validates that `input.ewtWithheld` matches the expected EWT total across allocated charges (within ₱1.00 tolerance for rounding).

### 3.4 Arrears Classification

```
FUNCTION classifyArrears(monthsInArrears, leaseRegime):
  IF monthsInArrears = 0:
    → CURRENT
  ELSE IF monthsInArrears = 1:
    → WARNING (display only, no alert)
  ELSE IF monthsInArrears = 2:
    → WARNING (create WARNING alert for controlled)
  ELSE IF monthsInArrears >= 3 AND leaseRegime = CONTROLLED_RESIDENTIAL:
    → EJECTMENT (create OVERDUE alert; RA 9653 Sec. 9(b) ground)
  ELSE IF monthsInArrears >= 3 AND leaseRegime != CONTROLLED_RESIDENTIAL:
    → OVERDUE (no statutory ejectment threshold for commercial)
```

### 3.5 Overpayment Handling

If `paymentAmount + ewtWithheld > totalOutstandingCharges`:
- All charges are fully satisfied
- `remainingCredit = paymentAmount + ewtWithheld - totalOutstandingCharges`
- The credit amount is **not** stored as a separate entity — it manifests as a **negative balance** in the TenantBalance materialized view
- On the next billing run (P5), the prior balance carry-forward will show the credit
- The UI displays credit balances in green with "(credit)" label
- No separate refund workflow — the admin applies credit to future charges automatically

### 3.6 Form 2307 Quarterly Tracking

```
FUNCTION computeExpected2307(tenantId, quarter):
  1. Find all payments from this corporate tenant in the quarter
     WHERE date_issued BETWEEN quarter_start AND quarter_end
       AND is_reversed = false
       AND ewt_withheld > 0
  2. SUM(ewt_withheld) = expected_amount
  3. Corporate tenants must submit 2307 within 20 days after quarter-end
     (RR 02-98 Sec. 2.58)
  4. If not received by deadline → create FORM_2307_MISSING alert

Reconciliation:
  - When 2307 is received: compare received_amount vs expected_amount
  - Tolerance: ±₱1.00 (rounding differences)
  - If |delta| > ₱1.00: is_reconciled = false, flag for follow-up
  - If |delta| ≤ ₱1.00: is_reconciled = true
```

## 7. Edge Cases

1. **Overpayment:** Cash received exceeds total outstanding. Credit balance stored as negative in TenantBalance. Applied automatically on next billing run. No separate refund workflow in MVP.

2. **Corporate EWT partial payment:** Corporate tenant pays ₱95,000 cash + ₱5,000 EWT withheld on ₱100,000 base (₱112,000 invoice total including VAT). The charge is satisfied at ₱100,000 (cash ₱95,000 + EWT ₱5,000 covers base). The ₱12,000 VAT is collected in cash as part of the ₱95,000. Reconciliation: cash (₱95,000) + EWT (₱5,000) = base + VAT portion of cash. Implementation: `allocatePayment` treats `cashReceived + ewtWithheld` as the total payment amount when computing allocation.

3. **Bounced check:** Admin uses `payment.reverse` with reason "Bounced check." All allocations are unwound, charge statuses restored. The OR number is preserved (BIR numbering is gapless — cannot void a number) but the receipt IssuedDocument is annotated as reversed. The Form 2307 expected amount is adjusted.

4. **Multiple charge types in one payment:** Automatic allocation per Art. 1253 pays penalties first, then Art. 1254 sorts remaining by most onerous (oldest overdue first). A single ₱50,000 payment might touch: 2 penalty charges, 1 rent, 1 water, partial electric.

5. **Consigned payments (RA 9653 controlled units):** Tracked via `payment_method = OTHER` with `remarks` describing consignation venue and reference. A consigned payment resets arrears counting. The system does not verify the consignation itself — that requires human review.

6. **Post-dated checks:** Recorded with `date_issued` = check date (not receipt date). `deposited_date` is set when the check clears. If the check bounces before deposit, admin reverses the payment.

7. **Payment during tacit reconduction:** Payment allocation continues normally. Month-to-month lease status does not affect the allocation algorithm. The 10-year prescription on each installment is tracked via `charge.due_date` (Art. 1144).

8. **Advance rent recognition:** Advance rent is a `Charge` created by P5 with a future `billing_period`. Payment against it is a normal allocation. No special deferred income tracking in MVP — the accountant handles journal entries.

9. **Zero-balance payment with EWT only:** A corporate tenant may "pay" by submitting only EWT documentation (₱0 cash + ₱5,000 EWT on a ₱5,000 base charge). This is valid — `amount` = 0 is rejected, but `amount` reflects cash only. The system should accept payments where `amount >= 0` as long as `amount + ewtWithheld > 0`. **Schema adjustment:** Change `chk_payment_positive` to `CHECK(amount >= 0 AND (amount + ewt_withheld) > 0)`.

10. **Stale TenantBalance after concurrent mutations:** The materialized view refresh runs inside the transaction. Two concurrent payments for the same tenant will serialize on the `REFRESH MATERIALIZED VIEW CONCURRENTLY` — the second refresh sees the first's committed data. No race condition.
