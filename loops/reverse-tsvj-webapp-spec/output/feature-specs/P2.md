# P2 — Water Billing

*Feature Spec | Depends on: F0 (Foundation) | Implementation order: 5 of 15 (MVP Wave 2: P2+P3+P4)*

---

## Summary

Convert the monthly Maynilad master meter bill plus individual sub-meter readings into per-tenant water charges using the legally mandated per-tier billing method (each tenant starts at the lowest bracket). Allocate common area water as a separate line item. Enforce the strict no-markup rule (total billed ≤ master bill). Differentiate sewerage charges by tenant type (commercial only). Support shared meters (co-tenancy) and defective meter estimated billing.

#### `water.saveReadings`

Batch save — upserts readings for all meters in one call. Used by the meter reading entry form.

```ts
input: z.object({
  billingPeriod: z.string().regex(/^\d{4}-\d{2}$/),
  readings: z.array(z.object({
    waterMeterId: z.number().int().positive(),
    readingDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
    readingValue: z.string().regex(/^\d+(\.\d{1,2})?$/),
    isEstimated: z.boolean().default(false),
  })).min(1),
})
```

Validation:
- Each meter must exist and be ACTIVE or DEFECTIVE
- `readingValue` must be non-negative
- If a reading already exists for this meter+period, overwrite it (upsert)
- Does NOT enforce reading ≥ previous (handled as warning in UI, confirmed by user)

#### `water.run`

Computes all water charges for a billing period. Transaction-wrapped.

```ts
input: z.object({
  propertyId: z.number().int().positive(),
  billingPeriod: z.string().regex(/^\d{4}-\d{2}$/),
  allocationMethod: z.enum(["FLOOR_AREA", "EQUAL_SPLIT"]),
})
```

**Preconditions checked (returns PRECONDITION_FAILED if not met):**
1. Maynilad bill exists for this property+period
2. Rate schedule exists with effective_date ≤ billing period start
3. All ACTIVE sub-meters have readings for this period
4. No FINALIZED run exists for this property+period (a DRAFT run is deleted and recreated)

**Process:**
1. Delete existing DRAFT run for this period (if any)
2. Fetch Maynilad bill, applicable rate schedule, all readings
3. For each SUB_METER with ACTIVE status and tenant assignment:
   a. Get current + previous readings; compute consumption delta
   b. Handle shared meters: compute full charge, then apply `share_pct`
   c. Call `@tsvj/computations` `computeWaterBilling()` with consumption and rate schedule
   d. Create `water_charge` row
4. For COMMON_AREA meters:
   a. Sum consumption, compute tiered cost (residential rates, no sewerage)
   b. Call `allocateCommonArea()` to split among active tenants
   c. Create `water_charge` rows with `common_area_share` populated
5. Sum total_billed, compute reconciliation_diff
6. Create `water_billing_run` with status=DRAFT
7. Return the run with all charges

**Returns `BAD_REQUEST` if:**
- Any meter reading has negative consumption (current < previous) and is NOT flagged as estimated
- Total billed > Maynilad total (over-recovery — must not happen)

#### `water.finalize`

```ts
input: z.object({
  waterBillingRunId: z.number().int().positive(),
})
```

**Process (transaction-wrapped):**
1. Verify run status = DRAFT (return `PRECONDITION_FAILED` if FINALIZED)
2. Verify total_billed ≤ Maynilad bill total_amount (reconciliation re-check)
3. For each water_charge in the run:
   a. Create a `charge` record: `charge_type_id = WATER` (or `COMMON_WATER` for common area shares), `base_amount = total_charge`, `vat_rate = 0.00`, `is_vat_exempt = true`
   b. Update `water_charge.charge_id` with the new charge ID
4. Update run: `status = FINALIZED`, `finalized_at = now()`, `finalized_by = currentUser`
5. Invalidate caches: `water.listRuns`, `charge.list`, `billingRun.list`

#### `water.deleteRun`

```ts
input: z.object({
  waterBillingRunId: z.number().int().positive(),
})
```

Returns `PRECONDITION_FAILED` if status = FINALIZED. Cascades delete to `water_charge` rows (ON DELETE CASCADE).

## 4. Business Logic

### 4.1 Computation Functions (from `@tsvj/computations`)

All water billing computation is delegated to the `@tsvj/computations` package. The tRPC router bridges DB data → computation input → computation output → DB persistence.

**Per-Tier Water Billing** (`computeWaterBilling`):

```
FOR each tier (ordered by minCuM ascending):
  tierConsumption = MIN(remaining, tier.maxCuM - tier.minCuM)
  tierCharge = roundWater(tierConsumption × ratePerCuM)
  remaining -= tierConsumption
  IF remaining ≤ 0: BREAK

basicCharge = SUM(tierCharges)
environmentalCharge = roundWater(basicCharge × environmentalChargePct)
sewerageCharge = isResidential ? 0 : roundWater(basicCharge × sewerageChargePct)
fcdaCharge = roundWater(consumption × fcdaPerCuM)
totalBeforeVAT = basicCharge + environmentalCharge + sewerageCharge + fcdaCharge + mscCharge
vatRate = "0.00"  (always — water pass-through is NOT VATable per BIR RR 16-2005)
vatAmount = "0.00"
totalCharge = totalBeforeVAT
```

Rounding: `roundWater` = HALF_UP to 2 decimal places.

**Common Area Allocation** (`allocateCommonArea`):

```
commonAreaCost = computeWaterBilling(commonAreaConsumption, residentialRates, noSewerage)
IF allocationMethod = FLOOR_AREA:
  share_i = roundWater(commonAreaCost.totalCharge × (floorArea_i / totalFloorArea))
IF allocationMethod = EQUAL_SPLIT:
  share_i = roundWater(commonAreaCost.totalCharge / activeTenantCount)
```

### 4.2 Business Rules Enforced in tRPC Router

| Rule | Enforcement | Source |
|------|-------------|--------|
| Per-tier billing (NOT blended) | Computation uses `computeWaterBilling` which processes tiers sequentially | MWSS IRR No. 2008-02 |
| No markup (total ≤ master) | `water.run` rejects if total_billed > maynilad_bill.total_amount | MWSS Charter RA 6234; RA 7581 |
| Sewerage = commercial only | `computeWaterBilling` checks `isResidential` flag (derived from `rentable.unit_type` via lease→lease_rentable→rentable) | Maynilad rate schedule |
| Common area = separate line | Common area charges stored with separate charge_type (COMMON_WATER) | MWSS regulations |
| Water = NOT VATable | All water charges created with `vat_rate = 0.00` | BIR RR 16-2005 |
| One reading per meter per period | UNIQUE constraint on `(water_meter_id, billing_period)` | Data integrity |
| DRAFT can be re-run | Existing DRAFT deleted before recomputing; FINALIZED cannot be re-run | Workflow |
| Finalization creates Charges | Only finalized runs create `charge` records visible to P5 billing | Billing integration |
| Shared meter split | Full charge computed first, then multiplied by `share_pct / 100` | Co-tenancy pattern |

### 4.3 Edge Cases

| Edge Case | Handling |
|-----------|---------|
| **Shared meters (co-tenancy)** | 4 cases at TSVJ. Compute full meter charge, then apply `share_pct` per co-tenant. Each co-tenant gets their own `water_charge` row. Sum of co-tenant charges = full meter charge (within rounding tolerance). |
| **Defective meter (estimated)** | `is_estimated = true` on reading. Use average of last 3 readings for this meter as estimated consumption. If < 3 readings exist, use the most recent non-estimated reading. Flag meter as DEFECTIVE in `water_meter.status`. |
| **Negative consumption** | `current_reading < previous_reading`. Two scenarios: (1) meter rollover → compute as `(max_reading_value - previous + current)`; (2) data entry error → reject in `water.run` unless `is_estimated = true`. Admin must confirm rollover via checkbox in reading entry. |
| **Implausible readings** | Consumption > 3× average of last 3 months. Flagged as warning in UI (not blocking). Admin confirms or corrects before running billing. |
| **Vacant units** | Meters with no active `water_meter_tenant` assignment. Reading entered for leak detection but no `water_charge` created. Consumption absorbed as landlord cost (appears in under-recovery). |
| **Zero consumption** | Valid — tenant away for the month. Still generates a `water_charge` with `total_charge = msc_share + common_area_share` (fixed components still apply). |
| **MSC split** | MSC from Maynilad bill split equally among all active tenant-assigned meters (not by consumption). `msc_share = roundWater(msc / activeTenantMeterCount)`. |
| **FCDA negative** | Quarterly FCDA can be negative (adjustment). Stored as negative `fcda_per_cu_m`, results in negative `fcda_charge` deduction. `total_charge` can be lower than `basic_charge` in rare cases. |
| **Rate schedule change mid-month** | Use the rate schedule effective on or before the billing period start. `water.run` selects `MAX(effective_date) WHERE effective_date <= billingPeriodStart`. |
| **New tenant mid-month** | First reading = move-in meter reading. No previous reading → consumption = current − move-in reading. MSC share pro-rated if specified in the billing run. |
| **Rounding variance in reconciliation** | Sum of rounded per-tenant charges may differ from Maynilad total by a few pesos due to rounding. Acceptable if `|reconciliation_diff| ≤ total_amount × 0.01` (1% tolerance). The diff is always "under-recovery" — never over. |

### 5.3 Rate Schedule Input

```ts
const rateScheduleInput = z.object({
  effectiveDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  tiers: z.array(z.object({
    minCuM: z.number().int().min(0),
    maxCuM: z.number().int().positive().nullable(),
    ratePerCuM: z.string()
      .regex(/^\d+\.\d{2}$/)
      .refine((v) => parseFloat(v) > 0, "Rate must be positive"),
  })).min(1).refine(
    (tiers) => {
      // Validate tiers are contiguous and non-overlapping
      for (let i = 1; i < tiers.length; i++) {
        if (tiers[i].minCuM !== (tiers[i-1].maxCuM ?? tiers[i-1].minCuM)) return false;
      }
      // Last tier must have null maxCuM (unlimited)
      return tiers[tiers.length - 1].maxCuM === null;
    },
    "Tiers must be contiguous with last tier unlimited"
  ),
  environmentalChargePct: z.string()
    .regex(/^\d\.\d{4}$/)
    .refine((v) => parseFloat(v) >= 0 && parseFloat(v) <= 1, "Must be 0–1"),
  sewerageChargePct: z.string()
    .regex(/^\d\.\d{4}$/)
    .refine((v) => parseFloat(v) >= 0 && parseFloat(v) <= 1, "Must be 0–1"),

  msc: z.string().regex(/^\d+\.\d{2}$/),
  notes: z.string().max(500).optional(),
});
```
