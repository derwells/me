# P12 — Tax Data Compilation

*Feature Spec | Depends on: F0 (Foundation), P5 (Monthly Billing — charges, invoices, VAT), P6 (Payment Tracking — payments, EWT, Form 2307), P11 (Rent Roll — VAT/2307 summaries, SAWT data), P14 (Expense Tracking — disbursements, supplier EWT, input VAT) | Implementation order: 14 of 15 (MVP Wave 4: P7+P12+P13+P14)*

---

## Summary

Compile structured tax data packages for BIR quarterly and annual filings across six sub-processes: Output VAT summary (2550Q), Income Tax data (1702Q/1702-RT), SAWT from received 2307 certificates, EWT withheld on suppliers (0619-E/1601-EQ/1604-E), DST register (Form 2000), and SLSP (Summary List of Sales/Purchases). Data flows from P5 (billing), P6 (payments), P11 (rent roll), and P14 (expenses) into read-only compilation views with XLSX/CSV export for accountant handoff. The property manager compiles the data; the external accountant uses it to file returns.

## 2. API Surface

### 2.1 tRPC Router — `tax`

All procedures use `protectedProcedure` (both Admin and Accountant can view tax data). Mutations that generate compilations use `adminProcedure`.

#### `tax.compileVAT`

```ts
input: z.object({
  quarter: z.string().regex(/^\d{4}-Q[1-4]$/),   // "2026-Q1"
})
output: z.object({
  compilationId: z.number(),
  data: vatSummaryDataSchema,
  summary: vatSummarySchema,
})
```

**Procedure: adminProcedure**

**Logic:**
1. Parse quarter into date range (e.g., "2026-Q1" → Jan 1 to Mar 31)
2. Query `issued_document` joined with `issued_document_line` + `charge` where `issue_date` within quarter range and `document_type = 'INVOICE'`
3. Aggregate:
   - `vatableSales` = SUM(base_amount) WHERE vat_rate = 0.12
   - `exemptSales` = SUM(base_amount) WHERE is_vat_exempt = true
   - `outputVat` = SUM(vat_amount) WHERE vat_rate > 0
4. Compute uncollected receivables adjustment (RR 3-2024):
   - Query charges from **prior quarter** with status NOT IN ('PAID') and days past due_date exceeding credit term (default 30 days from due_date)
   - `uncollectedAdjustment` = SUM(vat_amount) of qualifying uncollected charges
   - One-quarter claim window only — charges older than prior quarter are not eligible
5. Query `input_vat_record` (from P14) for current quarter:
   - `currentInputVat` = SUM(vat_amount) WHERE is_capital_good = false
   - `capitalGoodsInputVat` = SUM(vat_amount) WHERE is_capital_good = true
6. Get `carryForwardInputVat` from prior quarter's compilation (if exists, the negative vatPayable)
7. Compute totals:
   - `adjustedOutputVat` = outputVat - uncollectedAdjustment
   - `totalInputVat` = currentInputVat + capitalGoodsInputVat + carryForwardInputVat
   - `vatPayable` = adjustedOutputVat - totalInputVat
8. Build line items array from issued_document data
9. UPSERT `tax_compilation` with form_type='2550Q', period=quarter, data=result, summary=totals
10. Return compilation

#### `tax.compileIncomeTax`

```ts
input: z.object({
  quarter: z.string().regex(/^\d{4}-Q[1-4]$/),
  // Manual entries not derivable from system data
  manualEntries: z.object({
    otherIncome: z.string().optional(),        // Interest, misc income
    depreciation: z.string().optional(),       // From fixed asset register
    salaries: z.string().optional(),           // Staff payroll
    interestExpense: z.string().optional(),    // Loan interest
    insurance: z.string().optional(),          // Property insurance
    realPropertyTax: z.string().optional(),    // RPT paid
  }).optional(),
})
output: z.object({
  compilationId: z.number(),
  data: incomeTaxDataSchema,
})
```

**Procedure: adminProcedure**

**Logic:**
1. Parse quarter into date range
2. Revenue: query `charge` WHERE billing_period within quarter AND status IN ('FINALIZED', 'PAID', 'PARTIALLY_PAID')
   - `commercialRevenue` = SUM(base_amount) WHERE lease_regime = 'COMMERCIAL' or ('NON_CONTROLLED_RESIDENTIAL' AND rent > ₱15K)
   - `residentialRevenue` = SUM(base_amount) WHERE lease_regime in residential types
   - Add manualEntries.otherIncome (default "0")
3. Deductions: query `disbursement_voucher` (P14) WHERE date_paid within quarter
   - Group by expense_category to populate deduction breakdown
   - Add manual entries for categories not tracked in P14 (depreciation, salaries, insurance, interest expense, RPT)
4. Compute YTD cumulative by querying all prior quarters in same year + current quarter
5. Tax computation via `@tsvj/computations` tax module:
   - `computeIncomeTax({ ytdGrossIncome, ytdDeductions, rcitRate, yearOfOperations })`
   - Returns RCIT, MCIT (if applicable), taxMethod, taxDue
6. Credits: query `form_2307_record` WHERE is_reconciled = true for YTD EWT credits
7. Prior quarterly payments from earlier `tax_compilation` records with form_type='1702Q' for same year
8. Excess MCIT carry-forward from `app_settings.mcit_carry_forward` JSONB (manual tracking)
9. `balanceDue` = taxDue - ewtCredits - priorQuarterlyPayments - excessMcitCredits
10. UPSERT `tax_compilation`

#### `tax.compileSAWT`

```ts
input: z.object({
  quarter: z.string().regex(/^\d{4}-Q[1-4]$/),
})
output: z.object({
  compilationId: z.number(),
  data: sawtDataSchema,
})
```

**Procedure: adminProcedure**

**Logic:**
1. Query `form_2307_record` WHERE quarter = input.quarter AND status IN ('RECEIVED', 'DELIVERED', 'RECONCILED')
2. Join with `tenant` for TIN, billing_name
3. Build SAWT entries per RR 2-2006 format:
   - sequence: auto-increment starting at 1
   - payorTin: tenant.tin (first 9 digits)
   - payorBranchCode: tenant.tin (last 3 digits, default "000")
   - taxType: "IC" (Income — Creditable)
   - periodCovered: format as "MMYYYY-MMYYYY" (e.g., "012026-032026")
   - natureOfIncome: "Rental"
   - atcCode: "WC157"
   - taxBase: form_2307_record.received_amount ÷ 0.05 (back-compute gross from EWT, or use expected if received not yet available)
   - applicableRate: "0.05"
   - taxWithheld: form_2307_record.received_amount (or expected_amount if pending)
4. Aggregate totals
5. UPSERT `tax_compilation` with form_type='2550Q' (SAWT attaches to 2550Q and 1702Q — store under its own compilation)

**Note:** SAWT reuses data already tracked in P6's `form_2307_record` and P11's `getSAWTData`. This procedure formats it into the BIR-prescribed DAT file layout.

#### `tax.compileEWT`

```ts
input: z.object({
  quarter: z.string().regex(/^\d{4}-Q[1-4]$/),
})
output: z.object({
  compilationId: z.number(),
  data: ewtAgentDataSchema,
})
```

**Procedure: adminProcedure**

**Logic:**
1. Query `ewt_withheld_record` (P14) joined with `disbursement_voucher` and `supplier_payee` WHERE date_paid within quarter
2. Group by `supplier_payee` to build QAP entries:
   - Per payee: aggregate total income payment and total EWT withheld
   - Include ATC code from expense category
3. Monthly breakdown: group by month within quarter for 0619-E reconciliation
4. UPSERT `tax_compilation` with form_type='1601_EQ'

#### `tax.compileDSTRegister`

```ts
input: z.object({
  year: z.number().min(2020).max(2100),
})
output: z.object({
  compilationId: z.number(),
  data: dstRegisterDataSchema,
})
```

**Procedure: adminProcedure**

**Logic:**
1. Query `dst_computation` (P8) joined with `lease`, `tenant`, `lease_rentable` + `rentable` WHERE `created_at` year = input.year
2. Build register entries from existing DST data (P8 already computes DST)
3. Include filing status from `lease_execution_milestone` (DST_FILED milestone)
4. Aggregate totals (total DST, filed count, pending count)
5. UPSERT `tax_compilation` with form_type='FORM_2000'

#### `tax.compileSLSP`

```ts
input: z.object({
  quarter: z.string().regex(/^\d{4}-Q[1-4]$/),
})
output: z.object({
  compilationId: z.number(),
  data: slspDataSchema,
  salesCount: z.number(),
  purchasesCount: z.number(),
})
```

**Procedure: adminProcedure**

**Logic (per RR 1-2012):**
1. **Sales list:** Query `issued_document` (INVOICE type) + `issued_document_line` WHERE issue_date within quarter
   - Per tenant invoice: TIN, registered name, invoice number, date, gross, exempt, VATable, VAT
   - INSERT into `slsp_record` with record_type='SALES'
2. **Purchases list:** Query `disbursement_voucher` (P14) WHERE date_paid within quarter AND supplier_payee.is_vat_registered = true
   - Per purchase: supplier TIN, name, invoice number, date, gross, VAT
   - INSERT into `slsp_record` with record_type='PURCHASES'
3. UPSERT parent `tax_compilation` with form_type='SLSP'

All functions use `Decimal` from `decimal.js` with `HALF_UP` rounding to 2 decimal places for tax amounts.

### 3.2 Key Business Rules

1. **Accrual basis (RR 3-2024):** Output VAT declared at invoice issuance (P5 billing), not at payment receipt (P6). The 2550Q compilation aggregates from `issued_document.issue_date`, not `payment.date_issued`.

2. **Uncollected receivables (RR 3-2024, RMC 65-2024):** If a billed invoice remains unpaid past the credit term, output VAT is deductible from the **immediately following** quarter's 2550Q. One-quarter window only. 8 requisites must be met (see process catalog P12).

3. **SAWT from received 2307s (RR 2-2006):** Generated from `form_2307_record` with status RECEIVED/DELIVERED/RECONCILED. Only corporate tenants issue 2307s. Individual tenants do NOT — the system must not flag them as missing.

4. **EWT as withholding agent:** Separate from receiving 2307s. The corporation withholds EWT when paying suppliers (P14). QAP format groups by payee with ATC codes.

5. **DST register:** P8 already computes and tracks DST. P12 merely reads `dst_computation` records into a consolidated register view for the accountant.

6. **SLSP (RR 1-2012):** Filed quarterly alongside 2550Q. Sales list from invoices (P5), purchases list from disbursement vouchers (P14). Only VAT-registered entities on the purchases side.

7. **Income tax — RCIT vs MCIT:** From the 4th taxable year, whichever is higher applies. Excess MCIT carries forward 3 years as credit against RCIT only. Rate is 25% (or 20% for SMEs with net taxable ≤ ₱5M and assets ≤ ₱100M).

8. **Input VAT — capital goods:** 60-month amortization was sunset January 1, 2022 (TRAIN Law). Full input VAT on capital goods now claimable immediately in month of purchase regardless of amount (RMC 21-2022).

9. **Mixed-operation apportionment:** If the corporation earns both VATable and exempt income, input VAT is apportioned based on the ratio of VATable sales to total sales for the quarter.

## 7. Edge Cases

1. **No P14 data yet:** If P14 is not yet implemented, `tax.compileVAT` returns 0 for input VAT and `tax.compileEWT` returns empty QAP. Compilation still succeeds with partial data — the system should display a warning banner "Expense data not available — input VAT and supplier EWT sections are incomplete."

2. **Missing tenant TIN:** SAWT export requires TIN. If any corporate tenant has a null TIN, the compilation still runs but displays an AlertBanner warning and the SAWT entry shows "MISSING" in the TIN field. The export includes the entry for manual completion.

3. **2307 mismatch:** When `form_2307_record.reconciliation_delta` exceeds ±₱1, the entry is flagged as DISPUTED. The SAWT compilation uses `received_amount` (not `expected_amount`) for reconciled/received entries, and `expected_amount` for pending entries.

4. **Zero-EWT tenants:** Individual (non-corporate) tenants do not withhold EWT and should never appear in SAWT or 2307 tracking. The compilation filters by `tenant.is_corporate = true`.

5. **Uncollected receivables timing (RR 3-2024):** The one-quarter claim window starts when the credit term expires, not from invoice issuance. A Q1 invoice with 30-day terms unpaid on day 31 (still Q1) → adjustment claimable on Q2's 2550Q only. Q3 is too late.

6. **MCIT carry-forward expiry:** Excess MCIT from Year N offsets RCIT in Years N+1, N+2, N+3 only. After 3 years, unused excess expires. The system tracks per-year carry-forward in `app_settings.mcit_carry_forward` JSONB with expiry enforcement.

7. **Mixed-operation input VAT proration:** When both VATable and exempt income exist, input VAT is prorated by `vatableSales / totalSales`. The computation function handles this transparently, but the UI should show the apportionment rate alongside the creditable amount.

8. **Partial year operations:** If the corporation starts or stops mid-year, quarterly compilations cover only the operational period. Thresholds (e.g., ₱3M VAT registration) may need pro-rating. The system does not auto-detect this — the admin manually adjusts via notes.

9. **DST register read-only:** P12 only reads `dst_computation` from P8. It does not create or modify DST records. If P8 is not implemented, the DST register page shows empty state.

10. **SLSP with no VAT-registered suppliers:** The purchases list only includes transactions with VAT-registered suppliers (per RR 1-2012). If no supplier_payee has `is_vat_registered = true`, the purchases section is empty — this is correct behavior, not an error.

11. **Recompilation:** Running `tax.compileVAT` for a quarter that already has a compilation replaces the existing DRAFT record (UPSERT). REVIEWED or EXPORTED compilations are NOT overwritten — the admin must delete them first and recompile.
