# P7 — Security Deposit Lifecycle

*Feature Spec | Depends on: F0 (Foundation), P5 (Monthly Billing Generation), P6 (Tenant Payment Tracking) | Implementation order: 9 of 15 (MVP Wave 3: P8+P9+P10)*

---

## Summary

Track security deposits through the full lifecycle: collection at lease start (with regime-aware caps for controlled residential), bank-holding and interest accrual (mandatory for controlled leases per RA 9653 Sec. 7), itemized deductions at lease end, tax reclassification of applied/forfeited amounts (BIR Ruling 118-12), and refund with deadline monitoring. Generate `DEPOSIT_APPLICATION` charges that feed into P5 (billing) and P12 (tax reporting). Alert when refund deadlines are approaching or overdue.

### 1.3 Charge Type Seed — `DEPOSIT_APPLICATION`

Added to the `charge_type` seed data (supplements F0 seed):

| Code | Label | is_vatable | default_vat_rate | is_recurring |
|------|-------|:----------:|:----------------:|:------------:|
| DEPOSIT_APPLICATION | Security Deposit Applied | true | 0.12 | false |

VAT on deposit application follows the same rule as rent (V5 from cross-cutting extract):
- Commercial lease or residential rent > PHP 15K/month → 12% VAT
- Residential rent ≤ PHP 15K/month → 0% (exempt)

EWT: 5% on the applied amount when tenant is corporate (ATC WC157, per RR 02-98 Sec. 2.57.2(B)).

## 2. API Surface

### 2.1 Router — `apps/web/src/routers/deposit.ts`

All mutations are `adminProcedure`. All queries are `protectedProcedure`.

| # | Procedure | Type | Auth | Description |
|---|-----------|------|------|-------------|
| 1 | `deposit.list` | query | protected | List all security deposits with tenant/lease join, filterable by status, tenant, regime |
| 2 | `deposit.getById` | query | protected | Full deposit record with interest accruals, deductions, applications, refund |
| 3 | `deposit.getByLeaseId` | query | protected | Get deposit for a specific lease (returns null if none) |
| 4 | `deposit.create` | mutation | admin | Create deposit record at lease start (validates regime caps) |
| 5 | `deposit.recordInterest` | mutation | admin | Record monthly interest accrual from bank statement |
| 6 | `deposit.recordInterestBatch` | mutation | admin | Batch record interest for multiple deposits in one period |
| 7 | `deposit.initiateReturn` | mutation | admin | Begin return process: set turnover_date, compute refund deadline |
| 8 | `deposit.addDeduction` | mutation | admin | Add itemized deduction (validates against remaining balance) |
| 9 | `deposit.removeDeduction` | mutation | admin | Remove a deduction (before finalization only) |
| 10 | `deposit.previewRefund` | query | protected | Dry-run refund computation: shows deductions, interest, net refund, tax impact |

### 2.2 Procedure Details

**`deposit.list`**
```ts
input: z.object({
  status: z.enum(["HELD", "PARTIALLY_APPLIED", "FULLY_APPLIED", "REFUNDED"]).optional(),
  tenantId: z.number().int().positive().optional(),
  leaseRegime: z.enum(["CONTROLLED_RESIDENTIAL", "NON_CONTROLLED_RESIDENTIAL", "COMMERCIAL"]).optional(),
  cursor: z.number().int().positive().optional(),
  limit: z.number().int().min(1).max(100).default(20),
})
output: {
  items: Array<{
    id, leaseId, tenantId, amount, depositMonths, status, leaseRegime,
    depositedDate, refundDeadline, totalInterest, totalDeductions,
    tenant: { id, name, isCorporate },
    lease: { id, unitDesignation, status },
  }>,
  nextCursor: number | null,
}
```

**`deposit.getById`**
```ts
input: z.object({ id: z.number().int().positive() })
output: {
  ...securityDeposit,
  tenant: { id, name, tin, isCorporate },
  lease: { id, unitDesignation, status, leaseRegime, monthlyRent },
  interestAccruals: Array<{ id, accrualPeriod, interestRate, interestAmount, bankStatementRef }>,
  deductions: Array<{ id, description, category, amount, supportingDocUrl, chargeId }>,
  applications: Array<{ id, appliedAmount, applicationDate, vatAmount, ewtAmount, chargeId }>,
  refund: { id, refundAmount, interestReturned, method, scheduledDate, actualDate, isOverdue } | null,
  totalInterest: string,
  totalDeductions: string,
  netBalance: string,  // amount + totalInterest - totalDeductions
}
```

**`deposit.create`**
```ts
input: z.object({
  leaseId: z.number().int().positive(),
  depositMonths: z.number().int().min(1),
  bankRef: z.string().max(200).optional(),
  depositedDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional(),
  notes: z.string().max(500).optional(),
})
```
Logic:
1. Load lease with tenant — verify lease exists and is ACTIVE or DRAFT
2. Check no existing deposit for this lease (`UNIQUE` constraint backup)
3. Validate regime caps: if `CONTROLLED_RESIDENTIAL`, `depositMonths` must be ≤ 2 (RA 9653 Sec. 7)
4. Compute `amount = depositMonths × lease.monthly_rent`
5. Insert `security_deposit` with lease_regime snapshot
6. Return created deposit

**`deposit.recordInterest`**
```ts
input: z.object({
  securityDepositId: z.number().int().positive(),
  accrualPeriod: z.string().regex(/^\d{4}-\d{2}$/),
  interestRate: z.string().regex(/^\d+(\.\d{1,5})?$/),
  bankStatementRef: z.string().max(200).optional(),
})
```
Logic:
1. Load deposit — verify status is HELD or PARTIALLY_APPLIED
2. Compute `interestAmount = roundMoney(deposit.amount × Decimal(interestRate))`
3. Insert `deposit_interest_accrual` (unique constraint prevents duplicate period)
4. Return created accrual

**`deposit.recordInterestBatch`**
```ts
input: z.object({
  accrualPeriod: z.string().regex(/^\d{4}-\d{2}$/),
  entries: z.array(z.object({
    securityDepositId: z.number().int().positive(),
    interestRate: z.string().regex(/^\d+(\.\d{1,5})?$/),
    bankStatementRef: z.string().max(200).optional(),
  })).min(1).max(50),
})
```
Logic:
1. For each entry, validate deposit exists and is active
2. Compute interest per entry
3. Insert all accruals in a single transaction
4. Return `{ applied: number, skipped: Array<{ depositId, reason }> }`

**`deposit.initiateReturn`**
```ts
input: z.object({
  securityDepositId: z.number().int().positive(),
  leaseEndDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  turnoverDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
})
```
Logic:
1. Load deposit — verify status is HELD
2. Set `lease_end_date` and `turnover_date`
3. Compute `refund_deadline`:
   - **Controlled residential:** 1 month from `MAX(leaseEndDate, turnoverDate)` (RA 9653 Sec. 7)
   - **Commercial:** 30 days from `MAX(leaseEndDate, turnoverDate)` (jurisprudential benchmark)
4. Create LEASE_EXPIRY alert for refund deadline tracking (60/30/15 day warnings)
5. Return updated deposit

**`deposit.addDeduction`**
```ts
input: z.object({
  securityDepositId: z.number().int().positive(),
  description: z.string().min(1).max(500),
  category: z.enum(["UNPAID_RENT", "UNPAID_UTILITY", "ASSOCIATION_DUES", "DAMAGE_REPAIR", "OTHER"]),
  amount: z.string().regex(/^\d+\.\d{2}$/),
  supportingDocUrl: z.string().url().optional(),
  chargeId: z.number().int().positive().optional(),
  notes: z.string().max(500).optional(),
})
```
Logic:
1. Load deposit with existing deductions — verify status is HELD or PARTIALLY_APPLIED
2. Validate: `totalDeductions + newAmount ≤ deposit.amount + totalInterest` — deductions cannot exceed deposit + accrued interest
3. Insert `deposit_deduction`
4. Update deposit status to `PARTIALLY_APPLIED` if first deduction
5. Return created deduction

**`deposit.removeDeduction`**
```ts
input: z.object({ deductionId: z.number().int().positive() })
```
Logic:
1. Load deduction → load parent deposit
2. Verify deposit status is not REFUNDED or FULLY_APPLIED
3. Delete deduction
4. If no remaining deductions, revert deposit status to HELD
5. Return success

**`deposit.previewRefund`**
```ts
input: z.object({ securityDepositId: z.number().int().positive() })
output: {
  depositAmount: string,
  totalInterest: string,
  deductions: Array<{ description, category, amount }>,
  totalDeductions: string,
  interestReturned: string,     // ALL for controlled, per-contract for commercial
  refundAmount: string,         // deposit - deductions + interestReturned
  isForfeiture: boolean,        // deductions >= deposit + interest
  taxReclassification: {
    amount: string,             // Applied/forfeited amount becoming gross receipts
    vatAmount: string,          // 12% if VATable (BIR Ruling 118-12)
    ewtAmount: string,          // 5% if corporate tenant (RR 02-98)
    isVatExempt: boolean,
    isCorporate: boolean,
  },
  refundDeadline: string | null,
  warnings: string[],
}
```
Computation delegated to `@tsvj/computations` deposit module: `computeDepositRefund()`.

**`deposit.finalizeReturn`**
```ts
input: z.object({
  securityDepositId: z.number().int().positive(),
  notes: z.string().max(500).optional(),
})
```
Logic (transaction-wrapped):
1. Load deposit with all sub-records — verify status is HELD or PARTIALLY_APPLIED
2. Verify `lease_end_date` and `turnover_date` are set (initiateReturn must have been called)
3. Compute refund via `computeDepositRefund()` from `@tsvj/computations`
4. If `taxReclassificationAmount > 0`:
   a. Determine VAT: `determineVATRate({ chargeType: "DEPOSIT_APPLICATION", ... })` from billing module
   b. Compute VAT: `roundVAT(reclassificationAmount × vatRate)`
   c. Compute EWT: if corporate tenant, `roundEWT(reclassificationAmount × 0.05)`, else 0
   d. Create `Charge` with type `DEPOSIT_APPLICATION`, `base_amount` = reclassification amount, `vat_rate`, `vat_amount`, `total_amount = base + vat`, `status = 'FINALIZED'`
   e. Insert `deposit_application` linking to the Charge
5. If `refundAmount > 0`:
   a. Create `deposit_refund` with `scheduled_date = refund_deadline`, `actual_date = NULL`
6. Update deposit status: `FULLY_APPLIED` if refundAmount = 0, else `PARTIALLY_APPLIED` stays until disbursement
7. If controlled and refund_deadline exists: create Alert `category = 'LEASE_EXPIRY'` for refund deadline tracking
8. REFRESH MATERIALIZED VIEW CONCURRENTLY tenant_balance (the application charge affects balance)
9. Return finalize result with refund details

**`deposit.recordRefundDisbursement`**
```ts
input: z.object({
  depositRefundId: z.number().int().positive(),
  actualDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  method: z.enum(["CASH", "CHECK", "BANK_TRANSFER"]),
  referenceNumber: z.string().max(200).optional(),
  notes: z.string().max(500).optional(),
})
```
Logic:
1. Load refund → load parent deposit — verify refund exists and `actual_date IS NULL`
2. Set `actual_date`, `method`, `reference_number`
3. Update deposit status to `REFUNDED` (terminal state)
4. Dismiss any LEASE_EXPIRY alert related to this deposit's refund deadline
5. Return updated refund

**`deposit.getOverdueRefunds`**
```ts
input: z.object({
  cursor: z.number().int().positive().optional(),
  limit: z.number().int().min(1).max(100).default(20),
})
output: {
  items: Array<{
    depositId, leaseId, tenantId, amount, refundAmount, refundDeadline,
    daysOverdue: number,
    tenant: { id, name },
    lease: { id, unitDesignation, leaseRegime },
  }>,
  nextCursor: number | null,
}
```
Filters: `deposit_refund.actual_date IS NULL AND deposit_refund.scheduled_date < now()`.

## 4. Business Logic

### 4.1 Computation Functions (from `@tsvj/computations`)

All deposit computation is delegated to the `@tsvj/computations` deposit module. The tRPC router bridges DB data → computation input → computation output → DB persistence.

**Deposit Validation** (`validateDeposit`):
```
if leaseRegime === "CONTROLLED_RESIDENTIAL":
  maxDepositMonths = 2
  maxAdvanceMonths = 1
  if depositMonths > 2 → violation: "Max 2 months deposit (RA 9653 Sec. 7)"
  if advanceMonths > 1 → violation: "Max 1 month advance (RA 9653 Sec. 7)"
else:
  maxDepositMonths = Infinity (no statutory cap)
  maxAdvanceMonths = Infinity

depositAmount = roundMoney(depositMonths × monthlyRent)
advanceAmount = roundMoney(advanceMonths × monthlyRent)
```

**Deposit Refund** (`computeDepositRefund`):
```
totalDeductions = sum(deductions.map(d => d.amount))

// Interest return rules
if leaseRegime === "CONTROLLED_RESIDENTIAL":
  interestReturned = accruedInterest   // ALL returned (RA 9653 Sec. 7)
else:
  interestReturned = accruedInterest   // Per contract; default = return all

refundAmount = roundMoney(depositAmount - totalDeductions + interestReturned)
isForfeiture = refundAmount <= 0

// Tax reclassification (BIR Ruling 118-12)
taxReclassificationAmount = roundMoney(min(totalDeductions, depositAmount))
  // Only the deposit portion is reclassified, not interest

// VAT on reclassified amount
if isVatExempt (residential ≤ ₱15K):
  vatOnReclassification = "0.00"
else:
  vatOnReclassification = roundVAT(taxReclassificationAmount × 0.12)

// EWT on reclassified amount (corporate tenants only)
if isCorporate:
  ewtOnReclassification = roundEWT(taxReclassificationAmount × 0.05)
else:
  ewtOnReclassification = "0.00"
```

**Rounding:**
- `roundMoney(value)` — HALF_UP to 2 decimals for all deposit computations
- `roundVAT(value)` — HALF_UP to 2 decimals (same as billing)
- `roundEWT(value)` — HALF_UP to 2 decimals (same as billing)

### 4.2 Deposit Cap Enforcement

| Lease Regime | Max Deposit | Max Advance | Legal Basis | Enforcement |
|-------------|:-----------:|:-----------:|-------------|-------------|
| CONTROLLED_RESIDENTIAL | 2 months | 1 month | RA 9653 Sec. 7 | Zod validation + CHECK constraint on lease table |
| NON_CONTROLLED_RESIDENTIAL | No cap | No cap | Civil Code Art. 1306 | None |
| COMMERCIAL | No cap | No cap | Civil Code Art. 1306 | None |

Cap is enforced at **three levels**:
1. F0 lease form `.superRefine()` blocks creation with excess deposit months
2. `deposit.create` tRPC procedure validates before insert
3. `@tsvj/computations` `validateDeposit()` returns violation message

### 4.3 Interest Obligation

| Regime | Interest Required? | Source | Implementation |
|--------|:-----------------:|--------|----------------|
| CONTROLLED_RESIDENTIAL | Yes — ALL accrued | RA 9653 Sec. 7: "any and all interest...shall be returned" | Monthly admin entry from bank passbook |
| NON_CONTROLLED_RESIDENTIAL | Per contract | Civil Code Art. 1956 | Monthly admin entry if contract requires |
| COMMERCIAL | Per contract | Civil Code Art. 1306 | Monthly admin entry if contract requires |

Interest is entered manually (from bank statements) — the system does not auto-compute interest rates. Admin records the rate and amount each month. System computes `interest_amount = roundMoney(deposit.amount × interestRate)` as a convenience check.

### 4.4 Deduction Rules

| Rule | Implementation | Legal Basis |
|------|---------------|-------------|
| Itemized documentation required | `description` field required; `supporting_doc_url` encouraged | Art. 1170, 1173 (good faith) |
| Commensurate to pecuniary damage only | UI warning text; no system enforcement (subjective) | RA 9653 Sec. 7 |
| No automatic forfeiture | System requires explicit deduction entry per item | Art. 1306/1409 |
| Deductions cannot exceed deposit + interest | Validated in `addDeduction` mutation | Business logic |
| Normal wear and tear excluded | UI guidance text (cannot enforce programmatically) | Civil Code Art. 1657 |

### 4.5 Tax Reclassification at Application

When deposit is applied to deductions or forfeited, the applied portion becomes taxable income (BIR Ruling 118-12):

| Event | Tax Treatment | System Action |
|-------|--------------|---------------|
| Deposit applied to unpaid rent | Gross receipts, subject to VAT + EWT | Create DEPOSIT_APPLICATION Charge |
| Deposit applied to damage repair | Gross receipts, subject to VAT | Create DEPOSIT_APPLICATION Charge |
| Deposit forfeited (full) | All becomes gross receipts | Create DEPOSIT_APPLICATION Charge for full amount |
| Deposit refunded | NOT taxable (remains liability) | No charge created for refunded portion |

Journal entry at application:
```
DR  Security Deposit Liability     [applied amount]
    CR  Rental Income                  [applied amount]
```

### 4.6 Refund Deadline Rules

| Regime | Deadline | Trigger | Penalty for Delay |
|--------|---------|---------|-------------------|
| CONTROLLED_RESIDENTIAL | 1 month from MAX(lease_end, turnover) | `initiateReturn` sets dates | PHP 25K–50K fine + imprisonment (RA 9653); DHSUD blacklisting |
| NON_CONTROLLED_RESIDENTIAL | 30 days (jurisprudential) | `initiateReturn` sets dates | 6% p.a. legal interest from demand (Art. 2209, Nacar v. Gallery Frames) |
| COMMERCIAL | 30 days (jurisprudential) | `initiateReturn` sets dates | 6% p.a. legal interest from demand |

### 4.7 Business Rules Enforced in tRPC Router

| Rule | Enforcement | Source |
|------|-------------|--------|
| One deposit per lease | UNIQUE index on `security_deposit(lease_id)` + pre-check in `create` | Business logic |
| Controlled ≤ 2 months | Zod `.refine()` + `validateDeposit()` | RA 9653 Sec. 7 |
| Deductions ≤ deposit + interest | Sum check in `addDeduction` | Business logic |
| Cannot deduct after REFUNDED | Status check in `addDeduction` | Business logic |
| Cannot finalize without turnover | Null check on `turnover_date` in `finalizeReturn` | Business logic |
| Cannot disburse before finalize | Check refund record exists in `recordRefundDisbursement` | Business logic |
| Overdue refund detection | Daily pg_cron sets `is_overdue = true` on past-deadline refunds | RA 9653 (controlled); fiduciary duty (commercial) |
| Interest returned to tenant (controlled) | `computeDepositRefund` always includes full `accruedInterest` for controlled | RA 9653 Sec. 7 |

### 4.8 Edge Cases

| Edge Case | Handling |
|-----------|---------|
| **Vacancy decontrol (threshold crossing)** | If controlled unit re-leased above PHP 10K threshold (P1 escalation), new lease is NON_CONTROLLED_RESIDENTIAL. New deposit record created with new regime — no retroactive change to old deposit. |
| **Tacit reconduction (month-to-month)** | Original deposit carries forward. No new deposit collected. System keeps existing `security_deposit` with status HELD. Interest continues to accrue. |
| **Partial forfeiture** | When deductions < deposit + interest, refund = remainder. Only the deducted portion is reclassified as income. System creates two records: `deposit_application` for deducted amount + `deposit_refund` for remainder. |
| **Full forfeiture** | When deductions ≥ deposit + interest, `refundAmount = "0.00"`. Entire deposit reclassified. System creates `deposit_application` for full deposit amount. No `deposit_refund` record (or refund with amount 0). |
| **Corporate tenant EWT on application** | When deposit applied for corporate tenant, 5% EWT on reclassified amount. Creates corresponding entry in form_2307_record (P6 infrastructure) for the quarter of application. |
| **Zero interest (commercial, no contract clause)** | Commercial leases may have no interest accrual. System allows 0 interest entries. Refund = deposit − deductions + 0. |
| **Lease renewal with deposit carry-forward** | If lease renewed without top-up, old deposit stays HELD. Admin can manually create new deposit on new lease if top-up collected. Old deposit refund follows normal lifecycle. |
| **Deposit top-up on regime change** | If lease re-negotiated from controlled to commercial, admin may collect additional deposit months. New deposit record on the new lease. Old deposit goes through return process. |
| **Advance rent reclassification** | Substance-over-form test: if "deposit" is non-refundable or earmarked as last month's rent, it's taxable at receipt (BIR Ruling 118-12). System flags via `notes` field — admin determines characterization. |
| **Overdue refund with no disbursement** | Daily pg_cron checks `deposit_refund WHERE actual_date IS NULL AND scheduled_date < now()`. Sets `is_overdue = true`. Creates Alert with `category = 'LEASE_EXPIRY'`, `level = 'OVERDUE'`. |

- [ ] `security_deposit` table exists with correct columns — `pnpm --filter @tsvj/db drizzle-kit check` exits 0 and table has `lease_id`, `tenant_id`, `amount`, `deposit_months`, `monthly_rent_at_collection`, `bank_ref`, `deposited_date`, `lease_regime`, `status`, `lease_end_date`, `turnover_date`, `refund_deadline` columns
- [ ] `deposit_interest_accrual` table exists — table has `security_deposit_id`, `accrual_period`, `interest_rate`, `interest_amount`, `bank_statement_ref` columns
- [ ] `deposit_deduction` table exists — table has `security_deposit_id`, `description`, `category`, `amount`, `supporting_doc_url`, `charge_id` columns
- [ ] `deposit_application` table exists — table has `security_deposit_id`, `applied_amount`, `application_date`, `charge_id`, `vat_amount`, `ewt_amount` columns
- [ ] `deposit_refund` table exists — table has `security_deposit_id`, `refund_amount`, `interest_returned`, `method`, `scheduled_date`, `actual_date`, `is_overdue` columns
- [ ] Unique constraint on lease_id — INSERT two `security_deposit` rows with same `lease_id` returns unique violation error
- [ ] Unique constraint on accrual period — INSERT two `deposit_interest_accrual` rows with same `security_deposit_id` + `accrual_period` returns unique violation error
- [ ] Check constraint enforces positive amount — INSERT `security_deposit` with `amount = 0` returns check constraint violation
- [ ] Check constraint enforces amount = months × rent — INSERT `security_deposit` with `amount` not equal to `deposit_months * monthly_rent_at_collection` returns check constraint violation
- [ ] `deposit_status` enum has correct values — enum includes HELD, PARTIALLY_APPLIED, FULLY_APPLIED, REFUNDED
- [ ] `refund_method` enum has correct values — enum includes CASH, CHECK, BANK_TRANSFER
- [ ] `DEPOSIT_APPLICATION` charge type seeded — `charge_type` table has row with code `DEPOSIT_APPLICATION`, `is_vatable = true`, `default_vat_rate = 0.12`

### Refund Computation (Unit Tests)
- [ ] Controlled refund with all interest returned — `computeDepositRefund({ depositAmount: "17000.00", accruedInterest: "342.00", deductions: [{ amount: "8500.00" }, { amount: "2500.00" }], leaseRegime: "CONTROLLED_RESIDENTIAL" })` returns `refundAmount: "6342.00"` and `interestReturned: "342.00"` and `totalDeductions: "11000.00"`
- [ ] Full forfeiture detected — `computeDepositRefund({ depositAmount: "17000.00", accruedInterest: "342.00", deductions: [{ amount: "17500.00" }], leaseRegime: "CONTROLLED_RESIDENTIAL" })` returns `isForfeiture: true` and `refundAmount: "0.00"`
- [ ] Tax reclassification on applied portion — `computeDepositRefund({ depositAmount: "17000.00", accruedInterest: "342.00", deductions: [{ amount: "11000.00" }], leaseRegime: "CONTROLLED_RESIDENTIAL" })` returns `taxReclassificationAmount: "11000.00"`
- [ ] VAT on commercial reclassification — `computeDepositRefund({ ..., leaseRegime: "COMMERCIAL", isVatExempt: false })` with `taxReclassificationAmount: "11000.00"` returns `vatOnReclassification: "1320.00"` (12%)
- [ ] VAT exempt for controlled residential ≤ ₱15K — `computeDepositRefund({ ..., leaseRegime: "CONTROLLED_RESIDENTIAL", isVatExempt: true })` returns `vatOnReclassification: "0.00"`
- [ ] EWT for corporate tenant — `computeDepositRefund({ ..., isCorporate: true })` with `taxReclassificationAmount: "11000.00"` returns `ewtOnReclassification: "550.00"` (5%)
- [ ] No EWT for individual tenant — `computeDepositRefund({ ..., isCorporate: false })` returns `ewtOnReclassification: "0.00"`
- [ ] Zero deductions = full refund — `computeDepositRefund({ depositAmount: "17000.00", accruedInterest: "342.00", deductions: [], ... })` returns `refundAmount: "17342.00"` and `isForfeiture: false` and `taxReclassificationAmount: "0.00"`

### tRPC Procedures
- [ ] `deposit.create` creates deposit — POST with valid leaseId and depositMonths returns deposit object with computed amount
- [ ] `deposit.create` rejects controlled > 2 months — POST with `{ leaseId: controlledLeaseId, depositMonths: 3 }` returns `BAD_REQUEST` with message containing "RA 9653"
- [ ] `deposit.create` rejects duplicate for same lease — POST for lease that already has deposit returns error (unique constraint)
- [ ] `deposit.getById` returns full deposit with sub-records — GET returns deposit with interestAccruals, deductions, applications, refund, totalInterest, totalDeductions, netBalance
- [ ] `deposit.getByLeaseId` returns deposit or null — GET for lease with deposit returns deposit; for lease without deposit returns null
- [ ] `deposit.recordInterest` creates accrual — POST with valid deposit, period, rate returns accrual with computed interest_amount
- [ ] `deposit.recordInterest` rejects duplicate period — POST with same deposit + period returns error (unique constraint)
- [ ] `deposit.recordInterestBatch` applies multiple — POST with 3 entries returns `{ applied: 3, skipped: [] }`
- [ ] `deposit.initiateReturn` sets dates and deadline — POST returns deposit with lease_end_date, turnover_date, refund_deadline computed; controlled lease gets deadline = 1 month from turnover
- [ ] `deposit.addDeduction` creates deduction — POST with valid description, category, amount returns deduction and updates deposit status to PARTIALLY_APPLIED
- [ ] `deposit.addDeduction` rejects excess — POST with amount exceeding remaining balance (deposit + interest − existing deductions) returns `BAD_REQUEST`
- [ ] `deposit.removeDeduction` deletes deduction — DELETE returns success; if last deduction, deposit reverts to HELD
- [ ] `deposit.previewRefund` returns computation — GET returns full breakdown: depositAmount, totalInterest, deductions, totalDeductions, interestReturned, refundAmount, isForfeiture, taxReclassification with VAT and EWT
- [ ] `deposit.finalizeReturn` creates charge + refund — POST creates DEPOSIT_APPLICATION Charge with correct base_amount/vat_rate/total_amount, DepositApplication record, DepositRefund record with scheduled_date, and refreshes TenantBalance materialized view
- [ ] `deposit.finalizeReturn` rejects without turnover — POST for deposit where turnover_date IS NULL returns `PRECONDITION_FAILED`
- [ ] `deposit.finalizeReturn` creates form_2307_record for corporate — POST for corporate tenant's deposit creates/updates form_2307_record for the application quarter with expected_amount reflecting 5% EWT
- [ ] `deposit.recordRefundDisbursement` records date/method — POST updates refund with actual_date and method, sets deposit status to REFUNDED
- [ ] `deposit.recordRefundDisbursement` rejects already disbursed — POST for refund where actual_date IS NOT NULL returns `PRECONDITION_FAILED`
- [ ] `deposit.getOverdueRefunds` returns past-deadline — GET returns refunds where actual_date IS NULL and scheduled_date < now(), with daysOverdue computed
