# Lektor Build & Deploy Pipeline

## Overview

The Lektor site uses a two-repo model: source lives in `pymc-labs/pymc-labs-website-source` (private), built artifacts are force-pushed to `pymc-labs/pymc-labs.github.io` (public) which serves via GitHub Pages with a custom domain.

## Repository Layout

| Repo | Visibility | Default Branch | Purpose |
|------|-----------|----------------|---------|
| `pymc-labs-website-source` | Private | `master` | Source code, content, templates |
| `pymc-labs.github.io` | Public | `gh-pages` | Built static files only |

## GitHub Actions Workflows

Three workflows in `.github/workflows/`:

### 1. `main.yml` — Production Deploy ("github pages")

**Trigger:** Push to `master`
**Runner:** `ubuntu-latest`
**Status:** Active (last run: 2026-03-06, **failed**)

Pipeline steps:
1. Checkout source
2. Setup Python **3.8** (note: pixi.toml specifies 3.9, CI uses older 3.8)
3. `pip install -r ./requirements.txt` (note: no `requirements.txt` exists in current repo — pixi replaced it)
4. `lektor build --output-path ./site`
5. Deploy via `peaceiris/actions-gh-pages@v3` to `pymc-labs/pymc-labs.github.io`

Deploy action config:
- **personal_token:** `${{ secrets.GHPAGES_TOKEN }}` (PAT, not deploy key)
- **external_repository:** `pymc-labs/pymc-labs.github.io`
- **publish_branch:** `gh-pages`
- **force_orphan:** `true` (every deploy creates a single orphan commit — no history)
- **CNAME:** `www.pymc-labs.com` (written automatically to deploy target)
- **enable_jekyll:** `false` (`.nojekyll` file created)

**Critical issue:** The workflow references `requirements.txt` which no longer exists. The project migrated to Pixi (`pixi.toml` + `pixi.lock`) but the CI was never updated. This is likely why the last deploy run failed.

### 2. `deploy-previews.yml` — PR Preview Deploys

**Trigger:** Pull request to `master`
**Runner:** `ubuntu-latest`
**Status:** Active (no recent runs found)

Pipeline steps:
1. Same Python 3.8 + pip setup as production
2. `lektor build --output-path site/`
3. Deploy to **Netlify** via `nwtgck/actions-netlify@v1.1`

Netlify config:
- **production-deploy:** `false` (preview only)
- **alias:** `deploy-preview-${{ github.event.number }}` (deterministic URL per PR)
- **Secrets:** `NETLIFY_AUTH_TOKEN`, `NETLIFY_SITE_ID`
- **PR comments:** Enabled (overwrites on re-push)
- **Timeout:** 1 minute

**Same `requirements.txt` issue** — would also fail on any new PR.

### 3. `merge-schedule.yml` — Scheduled Auto-Merge

**Trigger:** PR events (opened/edited/synchronized) + hourly cron (`0 * * * *`)
**Action:** `gr2m/merge-schedule-action@v1`
**Merge method:** Squash
**Purpose:** Allows scheduling PR merges by adding a `/schedule` comment with a date. Runs hourly to check.

This workflow runs frequently and successfully (all recent runs are this workflow).

## Build System

### Local Development (Pixi)

Defined in `pixi.toml`:

```toml
[dependencies]
python = "3.9.*"
lektor = ">=3.3.9"
flask = "2.3.1.*"
requests = "2.29.0.*"
jinja2 = "3.1.2.*"
werkzeug = "2.3.0.*"
pygments = "*"
mistune = "0.8.4.*"

[tasks]
serve = "lektor server --host 0.0.0.0 --port 0"
build = "lektor build"
dev = "lektor server --host 0.0.0.0 --port 0"
```

**Platforms:** `osx-arm64`, `linux-64` (no Windows support)

From `pymc-labs.lektorproject`:

**Note:** `url` still points to `pymc-labs.github.io` not `www.pymc-labs.com`. This affects canonical URLs and sitemap generation.

### Build Output

- `lektor build` outputs to `./site/` (gitignored)
- Output is a flat directory of HTML + static assets
- No build-time optimization (no image compression, minification, or bundling)
- Sitemap generated by Lektor's built-in sitemap support (template at `templates/sitemap.xml`)

## Hosting & DNS

### Production: GitHub Pages
- **Serving repo:** `pymc-labs/pymc-labs.github.io`
- **Branch:** `gh-pages` (single orphan commit, force-replaced each deploy)
- **Custom domain:** `www.pymc-labs.com` (CNAME file in repo)
- **HTTPS enforced:** No (`https_enforced: false` per API)
- **Status:** Built and serving

### PR Previews: Netlify
- Used only for PR preview deploys (not production)
- Deterministic preview URLs: `deploy-preview-{PR#}--{site-name}.netlify.app`
- Requires `NETLIFY_AUTH_TOKEN` and `NETLIFY_SITE_ID` secrets

### DNS Flow
```
www.pymc-labs.com → GitHub Pages (pymc-labs.github.io) → static HTML
```

## Issues & Risks

### Critical
1. **Broken CI pipeline** — `requirements.txt` no longer exists. Both `main.yml` and `deploy-previews.yml` reference it. The last production deploy (2026-03-06) failed. The site is serving stale content from the last successful deploy.
2. **CI/local Python version mismatch** — CI uses Python 3.8, pixi.toml specifies 3.9. Both are EOL (3.8 EOL Oct 2024, 3.9 EOL Oct 2025).
3. **HTTPS not enforced** — GitHub Pages `https_enforced` is `false`. HTTP requests are not redirected to HTTPS.

### Moderate
4. **Canonical URL mismatch** — `.lektorproject` url is `https://pymc-labs.github.io/` but actual domain is `www.pymc-labs.com`. Affects sitemap, OG tags, and any absolute URL generation.
5. **Force-orphan deploys** — Every deploy destroys git history in the target repo. No rollback capability beyond re-running the build.
6. **PAT-based auth** — Deployment uses a personal access token (`GHPAGES_TOKEN`) rather than a deploy key or GitHub App. Token expiry or account changes break deploys.
7. **Outdated GitHub Actions** — Uses `actions/checkout@v2` (current: v4), `actions/setup-python@v2` (current: v5), deprecated `set-output` syntax, `peaceiris/actions-gh-pages@v3` (current: v4).

### Low
8. **No build optimization** — No image compression, CSS/JS minification, or asset hashing in the build pipeline.
9. **`serve` and `dev` tasks are identical** — Both run `lektor server --host 0.0.0.0 --port 0`, no differentiation.
10. **Merge-schedule runs hourly** — Consumes Actions minutes even when no scheduled merges are pending.

## Summary

The build/deploy pipeline is in a broken state. The project migrated from pip (`requirements.txt`) to Pixi for local development but never updated CI. Production and preview deploys both fail. The site continues serving because GitHub Pages serves the last successful deploy from the orphan `gh-pages` branch. The hosting setup is straightforward (GitHub Pages + Netlify previews) but has security gaps (no HTTPS enforcement) and operational risks (PAT auth, no deploy history, outdated Actions versions).
